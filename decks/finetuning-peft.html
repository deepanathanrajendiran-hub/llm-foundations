Roadmap · the one-sentence version

The whole module in a line

Base LM → SFT → Assistant → RLHF / DPO

PEFT is how we do the SFT step cheaply — it doesn't change what SFT does.

Keep this slide as a mental scaffold for the whole deck. SFT teaches the behavior; preference optimization (Module 08) refines which good answer is preferred. PEFT is orthogonal — it is a way to run the SFT step with a tiny fraction of the memory. People conflate "PEFT" with "a different objective"; it is not — same loss, fewer trainable parameters.

Why it matters

A base model knows text, not behavior

  • Pretraining → brilliant at predicting the next token
  • Ask it a question → it may continue with more questions
  • It learned the shape of text, not the behavior of a helpful assistant
  • Fine-tuning = the short, supervised second act that teaches behavior
The core motivation. A base model has read much of the internet and is great at one thing: next-token prediction. But behavior — answering rather than rambling, stopping at the right place, using the right format — is not in there yet. SFT shows the model thousands of (instruction → good response) pairs and nudges its weights to answer the way you want. This is the cheapest, highest-leverage post-training step.

Intuition

Med school → residency → sticky notes

  • Pretraining = a med-school grad who memorized every textbook
  • SFT = residency: shadow senior doctors on real cases until knowledge becomes bedside manner
  • PEFT = a cheat-sheet of sticky notes riding alongside the frozen brain

Same behavior change, a tiny fraction of the cost.

Peel the notes off → you get the original doctor back.

The analogy is worth lingering on because it pre-loads three later points. "Rewiring the whole brain" = full fine-tuning: expensive and risky — the doctor might forget anatomy (catastrophic forgetting). "Sticky notes" = a few million extra parameters that ride alongside frozen weights. "Peel them off" = LoRA adapters are separable; you can revert to the base model exactly. This is why PEFT resists forgetting: the base weights never move.

The two key facts to seed now: only the small detour (A then B) is trainable, and because B is initialized to zero, the detour contributes nothing at step zero. So training begins exactly at the base model and the update grows gradually. No heavy math here — that comes later.

How it works

SFT is just more next-token prediction

  • …but on curated prompt + ideal response pairs
  • …formatted with a chat template
  • Special role markers <|user|> / <|assistant|>, a BOS and an EOS token
  • Wrong template → quality silently collapses
SFT uses the exact same loss as pretraining — cross-entropy on next tokens — but the data is different: curated instruction/response pairs, wrapped in the chat template the model was aligned on. The chat template is the literal string scaffold: role markers, begin-of-sequence, and crucially end-of-sequence so the model learns when to stop. Use the wrong template and the model never sees the cue pattern it learned during alignment, so quality drops with no error message. This is a classic silent failure.

How it works · the key trick

Loss masking: score only the response

  • Compute cross-entropy on response tokens only, not the prompt
  • The prompt is context we condition on, not text to learn to generate
  • Set prompt-token labels to the ignore index -100 → zero gradient
  • Forget it → the model wastes capacity parroting questions; EOS gets noisy
This is the single most asked SFT implementation detail. We only want the model to learn to produce the response, so we mask the prompt: in PyTorch, set those label positions to -100, which cross_entropy skips. If you skip masking, the model spends capacity learning to generate the user's question, and its end-of-sequence behavior degrades. Always mask the prompt. We will see the exact code for this later.

How it works · the cost problem

Full fine-tuning is memory-hungry

  • Full FT updates every weight
  • Needs weights + gradients + two Adam moment buffers
  • A 7B model in mixed precision → often 60–80 GB

PEFT freezes the base and trains a tiny add-on.

The dominant PEFT method: LoRA.

The memory accounting is why PEFT exists. Adam keeps two buffers (first and second moment) per trainable parameter, on top of the weights and gradients — roughly 4× the model size in fp16. A 7B model balloons to 60–80 GB, beyond a single consumer GPU. PEFT freezes W₀ so there are no gradients or optimizer states for it; you only pay for the tiny add-on. LoRA is the dominant such method.

How it works · the low-rank idea

LoRA assumes ΔW is low-rank

  • The change a task needs, ΔW, has low intrinsic rank
  • Don't learn a full d×k update — factorize it
  • ΔW = B·A with B (d×r), A (r×k), rank r ≪ min(d,k)
  • Scale by α/r
x→ W₀x+ A r×k→ B d×r×α/r h
The conceptual leap behind LoRA: empirically the update a downstream task needs lives in a low-dimensional subspace, so its rank is small. Instead of learning the full d×k matrix (millions of params per projection), factorize ΔW into two skinny matrices B (d×r) and A (r×k). With r much smaller than d or k, you train r(d+k) numbers instead of dk. The α/r scale decouples the update's magnitude from the rank. A is random Gaussian; B is zero at init so ΔW starts at zero.

How it works · mechanism

How it works · the payoff

Merge → zero inference latency

  • ΔW = (α/r)BA is a plain matrix, same shape as W₀
  • Merge once trained: W = W₀ + (α/r)BA
  • Merged model = same architecture & FLOPs as the base → zero added latency
  • Or keep adapters separate & hot-swap per request → multi-adapter serving
This is the property that made LoRA win over adapters. Because the update is a plain matrix of the same shape as W₀, you can fold it in: W = W₀ + (α/r)BA. The merged model is byte-for-byte the same architecture as the base, so it adds zero inference latency — unlike bottleneck adapters which insert extra layers in series. Alternatively, keep the adapter separate and swap adapters per request, which underpins multi-adapter serving (S-LoRA). One base, many tasks.

Approaches · 1 of 12

Full fine-tuning baseline

What: update all weights on your task data. How: standard backprop, small LR (≈1e-5–2e-5), whole network. Solves: maximum expressiveness — best with lots of data and a big behavior shift.

Pros
  • Highest quality ceiling; handles large shifts
  • No architectural assumptions
Cons
  • Huge memory & storage — one full model per task
  • Forgets general abilities most easily

vs previous: this is the reference point — everything that follows trades a little of its ceiling for a fraction of its cost.

Read more ↗

Full FT can move the model anywhere in weight space, so it has the highest ceiling, but it is the most expensive and the most prone to catastrophic forgetting because every weight moves. You also store a separate full-size model per task, and you pay weights + grads + two Adam moment buffers (≈4× the model in fp16). Everything that follows is a way to get most of this quality for a fraction of the cost.

How it works · Full fine-tuning

Full fine-tuning — step by step

data(x, y) pairs → forwardcross-entropy → backpropgrad for every W → Adam2 buffers / param → all W updated
  1. Feed curated (instruction, response) pairs through the whole network.
  2. Compute next-token cross-entropy on the response, backprop to every weight.
  3. Adam keeps first + second moment buffers per trainable param → ≈4× model in fp16.
  4. Step the optimizer: a 7B model balloons to 60–80 GB and you store a full copy per task.
The memory accounting is why PEFT exists. Adam keeps two buffers (first and second moment) per trainable parameter, on top of weights and gradients — roughly 4× the model size in fp16. A 7B model balloons past a single consumer GPU. Because every weight moves, the model can drift far from the base, which is both why it has the highest ceiling and why it forgets the most.

Approaches · 2 of 12

SFT / instruction tuning the SFT stage

What: the objective, not a parameter trick — train on (instruction, response) pairs. How: loss masking (labels = -100 on the prompt), keep EOS, optionally pack multi-turn. Solves: turns a next-token predictor into an instruction-follower; runs with full FT or any PEFT below.

Pros
  • Cheapest, highest-leverage post-training step
  • ∼1k high-quality examples can teach style (LIMA)
Cons
  • Wrong template / EOS silently degrades quality
  • Teaches format & style well; new facts only weakly

vs previous: orthogonal to "how" — full FT is one way to run SFT; instruction tuning is SFT on a broad mixture so it generalizes to unseen instructions (FLAN, InstructGPT).

Read more ↗

Important framing: SFT is an objective, not a parameter-efficiency method. You can do SFT with full FT or any PEFT method on this list. Instruction tuning is SFT on a broad task mixture so the model generalizes to instructions it never saw — the FLAN / InstructGPT idea. LIMA showed ∼1,000 carefully curated examples can already produce a strong assistant, which shifted effort from scraping to curation.

How it works · SFT / instruction tuning

SFT / instruction tuning — step by step

prompt + response → chat templaterole markers, EOS → mask promptlabel = -100 → score response only
  1. Wrap each pair in the model's chat template — role markers, BOS, and crucially EOS so it learns to stop.
  2. Set prompt-token labels to -100 — the prompt is context we condition on, not text to generate.
  3. Compute cross-entropy on response tokens only; same loss as pretraining, restricted sum.
  4. For instruction tuning, repeat across a broad mixture → generalizes to unseen instructions.
SFT uses the exact same loss as pretraining — cross-entropy on next tokens — but the data is curated pairs wrapped in the chat template the model was aligned on. The masking (label -100) implements "score the response only" in code. Use the wrong template and the model never sees the cue pattern it learned during alignment, so quality drops silently. Always include EOS in the loss so the model learns to terminate.

Approaches · 3 of 12

Bottleneck adapters first PEFT

What: small trainable modules inserted between frozen Transformer sub-layers (Houlsby 2019). How: down-project → nonlinearity → up-project, with a residual; Pfeiffer uses one per layer. Solves: first way to fine-tune BERT-era models with <5% of params.

Pros
  • Tiny footprint; modular per task
  • Strong, well-studied results
Cons
  • Adds depth in series → real inference latency (no merge)
  • Changes the forward graph

vs previous: first method to freeze the base and train a tiny add-on instead of all weights — but unlike LoRA, adapters sit in series so they can't be folded away.

Read more ↗

Adapters were the first popular PEFT method. Each is a tiny MLP bottleneck (down, nonlinearity, up) with a residual, dropped between sub-layers. The key difference from LoRA, and why LoRA replaced it: adapters sit in series in the forward graph, so they cannot be folded into existing weights — they add real depth and therefore real latency at inference. LoRA's parallel low-rank branch is mergeable; adapters are not.

How it works · Bottleneck adapters

Bottleneck adapters — step by step

hd-dim activation → down Wdd → m, m ≪ d → nonlinearity → up Wum → d → + residual
  1. Take the activation between two frozen sub-layers (e.g. after attention).
  2. Down-project to a small bottleneck dim m, apply a nonlinearity, up-project back to d.
  3. Add a residual so the module starts near identity; only the adapter trains.
  4. Because the block sits in series, it adds depth → cannot be merged → real latency.
Walk the bottleneck: the adapter sees an activation, squeezes it through a low-dim hidden layer (down, nonlinearity, up), and adds the result back via a residual so at init it is close to the identity. Only the two small projections train. The structural cost is that this block is inserted serially into the forward graph, so it genuinely lengthens the computation — that serial placement is exactly what LoRA's parallel branch avoids.

Approaches · 4 of 12

Prefix tuning soft prompts

What: prepend trainable "virtual token" vectors to the keys & values at every layer (Li & Liang 2021). How: optimize continuous prefix vectors while the model stays frozen, steering attention. Solves: very few trainable params; good for generation.

Pros
  • Influences all layers, not just the input
  • Tiny parameter count
Cons
  • Trickier to train (often reparam'd via an MLP); eats context budget
  • Usually below LoRA on modern LLMs

vs previous: adapters add parameters inside the network; prefix tuning leaves all weights untouched and instead prepends learned virtual tokens — it conditions the frozen model rather than modifying it.

Read more ↗

Now the soft-prompt family. Prefix tuning learns continuous prefix vectors prepended to the keys and values at every layer — they don't occupy real input slots but they do consume context length. They steer attention without touching weights. The trade-off: harder to optimize (often reparameterized through a small MLP for stability) and usually below LoRA in quality on modern LLMs.

How it works · Prefix tuning (soft prompts)

Prefix tuning — step by step

prefix vectorstrainable, frozen LM → prepend to K, Vevery layer → attention attendsreal tokens + prefix → steered output
  1. Initialize a small set of trainable continuous vectors; the whole LM stays frozen.
  2. Prepend them to the keys and values at every layer (prompt tuning does input only).
  3. Real tokens attend over prefix + sequence, so the prefix biases attention.
  4. Backprop updates only the prefix — a few thousand to a few hundred-k params.
Use the Play / Step / Reset controls. Narrate: the LM is frozen; we learn continuous "virtual token" vectors that have no fixed word; prefix tuning prepends them to keys and values at every layer (prompt tuning is the input-only special case); real tokens then attend over prefix + sequence so the prefix steers attention; only the prefix trains. The cost is that the prefix consumes context length at every layer and these are trickier to optimize than LoRA.

Approaches · 5 of 12

Prompt tuning soft prompts

What: learn a soft prompt only at the input embedding layer (Lester 2021). How: prepend p trainable embedding vectors to the input; nothing else changes. Solves: the extreme of efficiency — a few thousand params per task.

Pros
  • Fewest params of any method
  • Trivially swappable per task
Cons
  • Needs a very large base to shine (≥10B)
  • Limited expressiveness; input-only

vs previous: the input-only special case of prefix tuning — even cheaper (no per-layer prefix), but its expressiveness drops and it only matches full FT at very large scale.

Read more ↗

Prompt tuning is the input-only special case of prefix tuning: just prepend p trainable embedding vectors to the input; the rest of the model is untouched. This is the most parameter-efficient method on the list — a few thousand numbers per task. Its famous catch, from the Lester et al. title "The Power of Scale": it only becomes competitive with full FT at very large scale (≥10B). At small/medium scale LoRA wins.

How it works · Prompt tuning

Prompt tuning — step by step

p soft embeddingstrainable → prepend at inputembedding layer only → frozen Transformer → task output
  1. Allocate p trainable embedding vectors (e.g. 20–100) at the input only.
  2. Concatenate them in front of the real token embeddings; everything downstream is frozen.
  3. The signal must propagate through the whole frozen stack from a single input nudge.
  4. That is why a large base is needed — only then is the input lever expressive enough.
Prompt tuning is the minimal soft prompt: p learnable vectors prepended at the input embedding layer, then a fully frozen Transformer. Because the only lever is at the very input, the model has to be large for that lever to be expressive enough — the "power of scale" result. Per task you store just those p vectors, so swapping tasks is trivial, but at small scale LoRA is materially better.

Approaches · 6 of 12

P-tuning v2 deep prompts

What: trainable prompts at every layer (like prefix tuning) so soft prompts work across sizes (Liu 2021). How: deep prompt vectors per layer + stabilized optimization — a universal alternative to FT for NLU. Solves: the "soft prompts only work at huge scale" limit.

Pros
  • Works across scales, unlike shallow prompt tuning
  • Still very parameter-light
Cons
  • More to tune than input-only prompts
  • Consumes context at every layer

vs previous: fixes prompt tuning's scale problem by going deep — prompts at every layer (like prefix tuning) plus stable optimization, so it matches full FT at base/medium sizes, not just 10B+.

Read more ↗

P-tuning v2 fixes prompt tuning's scale problem by going deep — prompts at every layer, like prefix tuning, with stabilized optimization. The headline: it matches full fine-tuning on many NLU tasks at base and medium model sizes, not just at 10B+. The trade-off versus input-only prompts is more to tune and it consumes context at every layer.

How it works · P-tuning v2

P-tuning v2 — step by step

per-layer promptsL × deep vectors → inject at every layernot just input → stabilized optimbetter init / LR → FT-level NLU
  1. Allocate trainable prompt vectors for each layer, not just the input embedding.
  2. Inject them at every layer so the steering signal is refreshed throughout depth.
  3. Use a stabilized optimization recipe so training is reliable across model sizes.
  4. Result: matches full FT on NLU at base/medium sizes — closing the scale gap.
The mechanism is "deep prompts": instead of relying on a single input nudge propagating through a frozen stack, P-tuning v2 places trainable prompt vectors at every layer so the steering signal is re-applied throughout depth. Combined with a stabilized optimization recipe, this is what lets soft prompts match full fine-tuning on NLU at base and medium sizes, not only at 10B+.

Approaches · 7 of 12

BitFit bias-only

What: train only the bias terms, freeze everything else (Ben Zaken 2021). How: unfreeze biases (sometimes final layernorm/classifier) and update those alone. Solves: a startlingly strong baseline with <0.1% of params on smaller models.

Pros
  • Almost no new params; dead simple
  • Great sanity-check baseline
Cons
  • Limited ceiling on big tasks/models
  • Scales poorly to large generative LLMs

vs previous: soft prompts add virtual tokens; BitFit adds nothing — it just unfreezes the existing bias parameters, the minimalist extreme of PEFT.

Read more ↗

BitFit is the minimalist's PEFT: train only the bias terms. It is mostly interesting as a proof of how little capacity adaptation actually needs — under 0.1% of parameters and still a strong baseline on smaller models. It scales poorly to large generative LLMs, so in practice it is a sanity-check baseline rather than a production method, but it is a great talking point about the low intrinsic dimensionality of adaptation.

How it works · BitFit

BitFit — step by step

all weights Wfrozen · unfreeze biases b<0.1% of params → backprop to b only → shifted activations
  1. Freeze every weight matrix; leave only the additive bias vectors trainable.
  2. Optionally also unfreeze the final layernorm / classifier head.
  3. Backprop touches only the biases — an additive shift of each layer's activations.
  4. Tiny capacity is enough on small models; it underfits big generative LLMs.
The whole trick: every weight matrix stays frozen and only the bias vectors (and sometimes the final layernorm/classifier) are unfrozen. A bias is a pure additive shift of the activations, so BitFit can only translate, not rotate, each layer's representation. That is why it is a beautiful demonstration of low intrinsic dimensionality but underfits when the task needs a larger, richer change.

Approaches · 8 of 12

LoRA — Low-Rank Adaptation the default

What: learn a low-rank update \( \Delta W = \frac{\alpha}{r}BA \) beside frozen weights (Hu 2021). How: A random Gaussian, B=0 at start (so \( \Delta W = 0 \)); applied to q/v (often k/o & MLP). Solves: train \( r(d{+}k) \) not \( dk \) params — and merge for zero added latency.

Pros
  • ∼0.1–1% trainable params; merge = no latency
  • Hot-swappable; resists forgetting
Cons
  • Assumes a low-rank update (can underfit big shifts)
  • Rank / alpha / target-module choices matter

vs previous: unlike adapters it runs in parallel and folds into W₀ → zero serving latency; unlike soft prompts it modifies the weight transform directly, so it works at any scale.

Read more ↗

\( h = W_0 x + \frac{\alpha}{r} B A x \), with \( r \ll \min(d,k) \).

LoRA is the default you reach for. It learns the low-rank update, initialized so ΔW=0 at the start. Typically applied to the attention query and value projections, often also k/o and MLP. Two superpowers: ∼0.1–1% trainable parameters (so the optimizer state shrinks dramatically) and mergeability for zero added inference latency. Its main limitation is the low-rank assumption — a genuinely high-rank shift can underfit, which motivates rsLoRA.

How it works · LoRA

LoRA — step by step

x → W₀xfrozen base path   Ar×k down → Bd×r up, B=0 init ×α/r h = sum
  1. Frozen path: W₀ still computes the base output W₀x; no gradients flow to it.
  2. Down-project: A maps x into a rank-r bottleneck (random Gaussian init).
  3. Up-project: B maps back to d; B=0 at init → \( \Delta W = BA = 0 \), so we start exactly at the base.
  4. Scale & sum: \( h = W_0 x + \frac{\alpha}{r}BAx \); merge later for zero added latency.

\( \Delta W = \frac{\alpha}{r}BA \), trainable params \( r(d{+}k) \ll dk \).

Approaches · 9 of 12

QLoRA — 4-bit base + LoRA memory frontier

What: fine-tune a 4-bit quantized frozen base with higher-precision LoRA adapters (Dettmers 2023). How: NF4 storage, double quantization, paged optimizers; matmuls in bf16. Solves: fit a 65B/70B fine-tune on one 48 GB GPU at near-full quality.

Pros
  • Huge models on one consumer/pro GPU
  • Quality close to 16-bit LoRA
Cons
  • Slower per step (dequantize on the fly)
  • Merging into 4-bit weights can lose quality

vs previous: LoRA shrank trainable params; QLoRA also shrinks the frozen base by storing it in 4-bit — train less, then store less.

Read more ↗

QLoRA pushes the memory frontier. Three ingredients: NF4 quantization (an information-theoretically optimal 4-bit datatype for roughly-Gaussian weights), double quantization (quantize the quantization constants, saving ∼0.4 bits/param), and paged optimizers (offload optimizer state to CPU RAM on spikes instead of OOM-crashing). Critically the matmuls happen in a bf16 compute dtype — storage is 4-bit, compute is 16-bit. Result: fine-tune a 65B/70B model on a single 48GB GPU at near-full quality.

How it works · QLoRA

QLoRA — step by step

base → NF44-bit storage → double-quantquantize the scales → dequant → bf16per block, on the fly → + LoRA adapterbf16, trainable → paged optimizer
  1. Quantize: store the frozen base in 4-bit NF4 (∼35 GB for 70B vs ∼140 GB fp16).
  2. Double-quant: quantize the per-block scale constants too → ∼0.4 bits/param saved.
  3. Dequant for math: each block is restored to bf16 on the fly — storage 4-bit, compute 16-bit.
  4. Train + page: gradients update only the bf16 LoRA adapter; paged Adam survives memory spikes.
Walk the three QLoRA ingredients on top of LoRA. NF4 stores the frozen base in 4 bits; double quantization compresses the quantization constants themselves; at compute time each block is dequantized to bf16, so the matmul precision is unchanged. Gradients only ever touch the bf16 adapter — the base is frozen and quantized, acting as a fixed function. Paged optimizers offload optimizer state to CPU on spikes so a 70B fine-tune fits on a single 48GB GPU.

Approaches · 10 of 12

DoRA & rsLoRA refinements

What: two surgical fixes to vanilla LoRA. DoRA (Liu 2024) splits each weight into a magnitude vector (one per column) + a direction, putting LoRA only on direction → closes much of the gap to full FT, especially at low rank. rsLoRA (Kalajdzievski 2023) scales by \( \alpha/\sqrt{r} \) instead of \( \alpha/r \) → high ranks no longer vanish.

Pros
  • Better quality, same serving story as LoRA
  • rsLoRA unlocks usable high ranks
Cons
  • DoRA adds a little training overhead
  • One more knob; gains are task-dependent

vs previous: vanilla LoRA underfits at low rank (→ DoRA) and its \( \alpha/r \) scaling makes high ranks vanish (→ rsLoRA's \( \alpha/\sqrt{r} \)); both keep LoRA's zero-latency merge.

Read more (DoRA) ↗ · rsLoRA ↗

Two surgical fixes. DoRA decomposes each weight into a magnitude vector (one value per output column) and a direction, applying LoRA only to the direction — this recovers much of full FT's quality, especially at low rank. rsLoRA changes scaling from α/r to α/√r: with α/r, larger r shrinks the per-direction contribution toward vanishing gradients so high ranks don't help; α/√r keeps the update's scale stable as r grows. Both keep LoRA's zero-latency merge story.

How it works · DoRA & rsLoRA

DoRA & rsLoRA — step by step

W = magnitude mper column × direction V/‖V‖LoRA here    rsLoRA: ×α/√rstable at high r
  1. DoRA: decompose \( W = m \cdot \frac{V}{\lVert V \rVert} \) — a magnitude vector and a unit direction.
  2. Learn the magnitude m directly; apply the LoRA low-rank update only to the direction V.
  3. This mirrors full-FT learning dynamics → closes much of the gap, especially at low rank.
  4. rsLoRA: swap the scale \( \alpha/r \rightarrow \alpha/\sqrt{r} \) so the per-direction contribution stays stable as r grows.

DoRA: \( W = m\,\frac{V}{\lVert V\rVert} \) · rsLoRA: \( \Delta W = \frac{\alpha}{\sqrt{r}}BA \).

DoRA's mechanism: every weight is rewritten as a magnitude (one scalar per output column) times a unit-norm direction. The magnitude is learned directly and LoRA is applied only to the direction, which better matches how full fine-tuning moves weights and recovers quality at low rank. rsLoRA is a one-line change to the scaling constant: dividing by √r rather than r keeps the effective update magnitude roughly constant as you raise rank, so higher ranks finally help. Both merge exactly like LoRA.

IA³ — learned scaling vectors few-shot

What: rescale internal activations with learned vectors — no matrices (Liu 2022, T-Few). How: three small vectors per layer multiply keys, values, and the FFN intermediate element-wise. Solves: even fewer params than LoRA, strong few-shot, foldable into weights → no extra cost.

  • Excellent few-shot (T-Few)
Cons

IA³ takes a different tack: instead of adding low-rank matrices, it learns three small vectors per layer that rescale the keys, values, and FFN intermediate activations element-wise. Only those vectors train, so it is even smaller than LoRA, and like LoRA the scalings fold into the weights for no extra inference cost. It shines in few-shot settings (the T-Few recipe) and on tiny datasets where LoRA might overfit. Trade-off: lower ceiling on big shifts and less tooling.

How it works · IA³

IA³ — step by step

keys k × lk values v× lv → FFN intermediate× lff → fold into weightsno extra cost
  1. Learn three per-feature vectors per layer: \( \ell_k, \ell_v, \ell_{ff} \).
  2. Multiply the keys, values, and FFN intermediate activations by them element-wise (a gate).
  3. Only these vectors train — far fewer params than LoRA's \( r(d{+}k) \) matrices.
  4. Being multiplicative scalings, they fold straight into the weights → zero inference overhead.

Rescale: \( k \rightarrow \ell_k \odot k,\;\; v \rightarrow \ell_v \odot v \).

The mechanism is multiplicative, not additive: IA³ learns three element-wise scaling vectors per layer that gate the keys, the values, and the FFN intermediate activations. Because a scaling vector has one number per feature (not an r×(d+k) matrix) it is even lighter than LoRA, and since scaling commutes with the linear layers it folds into the weights with no inference overhead. This is what makes it the go-to for few-shot and tiny datasets.

  • One base serves many tasks/tenants
  • High throughput across mixed adapters
Cons

Read more ↗

S-LoRA is the deployment side of LoRA. Instead of merging, you keep adapters separate in a unified paged memory pool, page them between GPU and host on demand, and use custom batched kernels so requests targeting different adapters share a single base forward pass. This makes per-customer or per-task fine-tunes economical: one base model, thousands of adapters. The cost is a small per-token overhead from staying unmerged and real systems complexity.

How it works · S-LoRA

S-LoRA — step by step

1 base on GPUfrozen, shared → paged adapter poolthousands, GPU↔host → batched kernelsshared base matmul → per-request ΔW
  1. Keep one frozen base in GPU memory; store many small adapters in a unified paged pool.
  2. Page adapters between GPU and host on demand — only the active ones occupy fast memory.
  3. A custom batched kernel computes the shared base matmul once for the whole batch.
  4. Then applies each request's own low-rank \( (\alpha/r)BAx \) — mixed adapters, one forward pass.
The systems idea: amortize the expensive shared base compute across every request in the batch, regardless of which adapter each uses. One frozen base lives on the GPU; adapters live in a unified paged memory pool and are paged in/out like virtual memory. A custom gathered/batched kernel does the base matmul once and then applies each request's own ΔW, so thousands of adapters share one forward pass — the basis of multi-tenant LoRA serving.

The math

SFT loss — response tokens only

$$ \mathcal{L} = -\sum_{t \in \text{response}} \log p_\theta(y_t \mid y_{<t}, x) $$

\(x\) = prompt/instruction (conditioned on, never scored); \(y_t\) = the \(t\)-th response token; \(y_{<t}\) = response tokens before it; \(p_\theta\) = model's predicted next-token probability. The sum runs only over response positions — prompt tokens are masked (label -100) so they give no gradient.

Walk through it slowly. It is the ordinary autoregressive cross-entropy, but the summation index is restricted: t ranges over response positions only. x is the prompt — we condition on it but never score it. y_{<t} is teacher-forced, the true previous response tokens. The masking (label -100) is what implements "t in response" in code. Stress that this is the same loss as pretraining, just on different data with a restricted sum.

The math · LoRA

The LoRA forward pass

$$ h = W_0 x + \Delta W x = W_0 x + \frac{\alpha}{r} B A x $$

\(W_0 \in \mathbb{R}^{d\times k}\) = frozen pretrained weight; \(B \in \mathbb{R}^{d\times r}\) and \(A \in \mathbb{R}^{r\times k}\) = low-rank factors, rank \(r \ll \min(d,k)\); \(\alpha\) = fixed scaling constant; \(h\) = layer output.

At init \(A \sim \mathcal{N}(0,\sigma^2)\) and \(B=0\), so \(\Delta W = BA = 0\) — training starts exactly at the base model and the update grows gradually.

This is the equation to know cold. h is the sum of the frozen path W₀x and the learned detour (α/r)BAx. Emphasize the init asymmetry: A is random Gaussian (so gradients can flow), B is zero (so the product BA is zero at the start). If you initialized both randomly you would perturb the base model before any learning. Because BA starts at zero, the first forward pass equals the base model exactly.

The math · parameter count

Why it's so cheap

$$ \underbrace{r(d+k)}_{\text{LoRA}} \;\;\ll\;\; \underbrace{d\,k}_{\text{full update}} $$

\(r(d+k)\) = params in A (\(r\times k\)) plus B (\(d\times r\)); \(dk\) = a full dense update. For a square 4096×4096 projection at r=8: \(8(4096+4096)=65{,}536\) vs \(4096^2 \approx 16.8\)M — about 0.39%.

The savings come from replacing a quadratic dk with a linear r(d+k). For a 4096×4096 projection at rank 8, that is 65,536 trainable numbers versus 16.8 million — 0.39%. Multiply across all q/v projections in a 7B model and you go from billions of trainable params to a few million, which is why the Adam optimizer state shrinks from tens of GB to under a gigabyte. That memory drop is the whole point.

The math · scaling

α/r — and rsLoRA's α/√r

$$ \Delta W = \frac{\alpha}{r}\,BA \qquad\text{vs.}\qquad \Delta W_{\text{rs}} = \frac{\alpha}{\sqrt{r}}\,BA $$

The factor \(\alpha/r\) decouples the update's magnitude from the rank, so changing r doesn't force re-tuning the learning rate.

But with α/r, larger r shrinks the per-direction contribution and stalls learning; rsLoRA uses \(\alpha/\sqrt{r}\) so the scale stays stable as r grows.

The α/r factor is a normalization so that tuning r and tuning the learning rate are decoupled — handy in practice. But it has a side effect: as r grows, dividing by r shrinks each direction's contribution toward zero, so high ranks barely move validation loss. rsLoRA's fix is to divide by √r instead, which keeps the effective update scale roughly constant as r grows, letting larger ranks actually help. Use it only when you genuinely need high rank.

In code

A LoRA linear layer

import torch, torch.nn as nn

class LoRALinear(nn.Module):
    def __init__(self, base: nn.Linear, r=8, alpha=16):
        super().__init__()
        self.base = base                       # frozen pretrained Linear (d_out x d_in)
        for p in self.base.parameters():       # freeze W0 (and bias)
            p.requires_grad = False
        d_out, d_in = base.weight.shape
        self.scale = alpha / r                 # the alpha/r factor
        self.A = nn.Parameter(torch.empty(r, d_in))   # r x k
        self.B = nn.Parameter(torch.zeros(d_out, r))  # d x r, init 0 -> dW = 0
        nn.init.normal_(self.A, std=1.0 / r)          # A ~ N(0, sigma^2)

    def forward(self, x):
        # frozen path + low-rank path:  h = W0 x + (alpha/r) B A x
        return self.base(x) + self.scale * (x @ self.A.t() @ self.B.t())
Read the code as the equation made literal. Line by line: freeze every base parameter (no gradients); compute scale = alpha/r; A is a learnable r×d_in tensor initialized random Gaussian; B is d_out×r initialized to zeros so ΔW=0 at step zero. In forward, return the frozen path base(x) plus scale times x @ Aᵀ @ Bᵀ. Note B requires_grad is True by default since it's a fresh Parameter — only the base is frozen. This is exactly what the mechanism animation showed.

In code · loss masking

Mask the prompt with -100

import torch.nn.functional as F

IGNORE = -100  # cross_entropy skips these positions

def build_labels(input_ids, prompt_len):
    labels = input_ids.clone()
    labels[:, :prompt_len] = IGNORE     # mask the instruction tokens
    return labels

# logits: (batch, seq, vocab); labels: (batch, seq) with IGNORE on the prompt
def sft_loss(logits, labels):
    # shift so position t predicts token t+1
    logits = logits[:, :-1, :].reshape(-1, logits.size(-1))
    labels = labels[:, 1:].reshape(-1)
    return F.cross_entropy(logits, labels, ignore_index=IGNORE)  # only response counts
Two things to call out. First, build_labels sets the prompt span to -100 so the loss ignores it — this is loss masking in code. Second, the shift: logits at position t predict token t+1, so we slice logits[:, :-1] against labels[:, 1:]. ignore_index=-100 tells cross_entropy to skip the masked positions entirely. Get the shift wrong and you train the model to predict the current token from itself — a subtle, common bug.

Worked example · Part A

Parameter savings for one projection

One query projection at d = k = 4096, LoRA rank r = 8:

$$ \text{full} = d\,k = 4096 \times 4096 = 16{,}777{,}216 \;\approx\; 16.78\text{M} $$

$$ \text{LoRA} = r(d+k) = 8 \times (4096 + 4096) = 65{,}536 $$

LoRA trains 65,536 params vs 16.78M — about 0.39%. Across all q/v projections of a 7B model that's billions → a few million trainable params, so Adam's two buffers shrink from tens of GB to under a gigabyte.

Do the arithmetic live. Full update of a 4096×4096 matrix is 4096² ≈ 16.78M parameters. LoRA at rank 8 is 8×(4096+4096) = 65,536. The ratio is 0.39%. The punchline isn't just fewer weights — it's that Adam keeps two moment buffers per trainable parameter, so shrinking trainable params from billions to millions is what collapses the optimizer-state memory from tens of GB to under a GB.

Worked example · Part B

70B QLoRA on a 48 GB GPU

  • 4-bit base weights: 70B × ~4 bits ≈ 35 GB — dominant, and frozen (no grads/optimizer)
  • LoRA adapters + grads + Adam: a few tens of M bf16 params → under ~1 GB
  • Paged optimizer: offloads state to CPU on spikes instead of OOM-crashing
  • Activations: the rest; gradient checkpointing recomputes them to save memory

The trick: those 35 GB of frozen 4-bit weights would normally also demand grads + optimizer state (past 200 GB for full FT) — QLoRA trains only the tiny adapters on top.

This is the canonical "estimate the memory budget" interview question. The dominant term is the 35 GB of 4-bit frozen base weights — and because it is frozen, it carries no gradients or optimizer state. The adapters plus their gradients and Adam buffers are under a GB. Paged optimizers prevent OOM on spikes by offloading to CPU RAM. Gradient checkpointing recomputes activations in the backward pass to keep the activation footprint small. Contrast: full FT of 70B would push past 200 GB because every weight needs grads + two Adam buffers.

When to use what · the knobs

Two more rules

  • Many tasks from one base? → keep adapters unmerged, serve S-LoRA-style; merge only to ship one dedicated model
  • LoRA underfits at low rank? → DoRA · higher rank not helping? → rsLoRA (α/√r)
  • Need new facts, not new style? → consider RAG instead of (or with) fine-tuning
The last bullet is the one people get wrong most often. Fine-tuning teaches format and behavior far better than it injects fresh knowledge. If your real need is "the model should know my company's docs," RAG is usually the right tool — retrieve the facts at inference rather than trying to bake them into weights. You can combine the two: RAG for knowledge, a light SFT/LoRA for the house style and format.

Pitfalls

People think X, but actually Y

⚠ Pitfall
  • "Fine-tuning is how you add knowledge." → mostly no — SFT is great at style/format/behavior, unreliable for new facts. Use RAG for knowledge.
  • "LoRA adds inference latency." → only if unmerged; merge W = W₀ + (α/r)BA → zero added latency.
  • "The chat template is cosmetic." → wrong template silently tanks quality.
First three misconceptions. Knowledge: SFT shifts behavior, not facts — reach for RAG when you need fresh knowledge. Latency: unmerged LoRA adds a tiny overhead, but once merged the model is identical in shape to the base, so zero added latency — say this clearly, it is a frequent interview gotcha. Template: it is not decoration; the model was aligned on an exact role-marker/BOS/EOS pattern and silently degrades if you change it.

Pitfalls · continued

Four more traps

⚠ Pitfall
  • "Skip loss masking, it's fine." → without it the model parrots prompts & EOS degrades — always set prompt labels to -100.
  • "Higher LoRA rank is always better." → with α/r more rank barely moves val loss; use rsLoRA if you truly need high r.
  • "QLoRA's compute dtype = storage dtype." → no — stored 4-bit NF4, computed in bf16; misconfigure → NaNs.
  • "PEFT and full FT forget equally." → PEFT resists forgetting more; base weights stay frozen.
Four more. Loss masking: non-negotiable; without it capacity leaks into generating the prompt. Rank: with standard α/r, more rank often just slows training without helping — a sign the update is already low-rank; only go high-rank with rsLoRA. QLoRA dtype: storage is 4-bit NF4 but compute is bf16; conflating them causes instability or NaNs. Forgetting: PEFT freezes the base, so only a small add-on moves — that is why it resists catastrophic forgetting better than full FT.

Practice

Practice — click a question to reveal the answer

Full bank: 07 · SFT, Fine-tuning & PEFT — from the answer key, condensed.

  • What is SFT, and how does it differ from pretraining in objective and data? Easy▸ reveal
    Same next-token cross-entropy as pretraining, but on small curated (instruction, response) pairs with the prompt loss-masked. Pretraining is self-supervised on massive raw text for general structure; SFT shapes behavior, format, and instruction-following. It's the same loss function, not a new one — only the data (and masking) change.
  • Full fine-tuning vs PEFT — what are the main motivations for PEFT? Easy▸ reveal
    Full FT updates all weights; PEFT freezes the base and trains a small added/selected set. Motivations: far less GPU memory (no grads/optimizer state for billions of params), tiny MB checkpoints, easy adapter hot-swapping, and less catastrophic forgetting. PEFT doesn't always match full FT — big distribution shifts or knowledge-heavy tasks can still favor full FT.
  • Fine-tuning vs RAG for injecting new knowledge — when each, when both? Med▸ reveal
    RAG retrieves facts at inference — best for large, changing, citable corpora, updatable without retraining. Fine-tuning bakes in style, format, and behavior but is poor for volatile facts. Hybrid is common: fine-tune for tone/format, RAG for the actual facts. Fine-tuning is a lossy, stale-prone way to memorize many facts — don't rely on it for knowledge.
  • Explain loss masking on prompt tokens. Why compute loss only on the response? Med▸ reveal
    Set prompt/instruction labels to -100 so cross-entropy ignores them; we want the model to learn prompt→response, not to generate the user's instruction. Scoring the prompt wastes capacity and dilutes the response signal. Masked tokens are NOT removed from context — they still attend; only their loss contribution is zeroed.
  • What is a chat template, and why does the wrong one silently degrade quality? Med▸ reveal
    The model-specific format wrapping roles and turns — markers like <|user|>/<|assistant|> plus BOS/EOS that the model was aligned on. A wrong template puts tokens in unfamiliar positions and blurs prompt/response boundaries, hurting quality with no error. Train and serve templates must match exactly; mismatches fail silently.
  • How do special tokens (BOS/EOS/role markers) affect SFT, and what breaks if EOS is wrong? Med▸ reveal
    They delimit turns and signal boundaries; the model learns to start, stop, and switch roles on them. If EOS isn't appended (or is masked from the loss), the model never learns to stop — it generates to max tokens. Fix: append EOS and include it in the loss. Forgetting EOS in the loss-computed tokens is the classic "never stops generating" bug.
  • Explain how LoRA works — what it injects and why it cuts trainable params. Easy▸ reveal
    For frozen W (d×k) it learns a low-rank update ΔW = BA (A is r×k, B is d×r, r≪min(d,k)); the layer computes h = Wx + (α/r)BAx. Only A and B train, so params drop from d×k to r(d+k). W stays frozen — LoRA adds a parallel low-rank term, it doesn't replace W.
  • Write the LoRA forward pass and explain why one of A, B is initialized to zero. Med▸ reveal
    h = Wx + (α/r)·B·A·x, W frozen, A random (Gaussian/Kaiming), B=0. With B=0, ΔW=0 at start so the model begins exactly as pretrained; A's first-step gradient is 0 but B's is nonzero, so training still moves. Both can't be zero — then ΔW stays 0 forever and A never gets a gradient; exactly one (conventionally B) is zero.
  • Why does LoRA add no inference latency once merged, and what happens in the merge? Easy▸ reveal
    Merging computes W' = W + (α/r)BA and stores one matrix of the original shape, so the forward pass is a single matmul with no extra adapter compute — essentially zero added latency. You can un-merge by subtracting it back. Unmerged LoRA is NOT free — it adds a small per-layer BAx matmul until merged.
  • You raise rank from 8 to 64; val loss barely moves but training is slower. What does this tell you? Med▸ reveal
    The task's adaptation has low intrinsic rank — r=8 already captures it, so extra capacity adds only cost. Revert to the lower rank; the bottleneck is elsewhere: improve data, tune LR/epochs, expand target modules, or accept the base ceiling. Flat returns mean you do NOT need more rank — don't keep raising r.
  • What is QLoRA, and what core problem does it solve vs standard LoRA? Easy▸ reveal
    It quantizes the frozen base to 4-bit (NF4) and trains higher-precision LoRA adapters on top. Standard LoRA keeps the base in fp16/bf16, which dominates memory; QLoRA's 4-bit base cuts weight memory ∼4×, fitting a 65–70B fine-tune on one 48 GB GPU. Key tricks: NF4, double quantization, paged optimizers. QLoRA quantizes the base, not the adapters — adapters stay higher-precision.
  • Explain NF4 quantization — why is it well-suited to neural-net weights? Hard▸ reveal
    NF4 (4-bit NormalFloat) places its 16 levels at the quantiles of a standard normal, so each bin carries roughly equal probability mass — information-theoretically optimal for zero-centered Gaussian data. NN weights are approximately zero-mean Gaussian, so NF4 beats uniform int4/fp4 at the same bit width (applied block-wise). NF4 is a quantile-based, non-uniform scheme — not just uniform 4-bit.
  • What do rsLoRA and DoRA fix vs vanilla LoRA, and how do they reparameterize? Hard▸ reveal
    rsLoRA: vanilla's α/r scaling over-shrinks updates at high rank, capping the benefit of more rank — it scales by α/√r so per-rank contribution stays stable. DoRA: decomposes each weight into magnitude + direction, applying LoRA to the direction — better matches full-FT dynamics, especially at low rank. Don't conflate them: rsLoRA fixes high-rank scaling; DoRA decomposes magnitude/direction.
  • Catastrophic forgetting in fine-tuning — why is PEFT often more resistant? Med▸ reveal
    Forgetting is the loss of general capabilities when fine-tuning overwrites pretrained weights on narrow data. PEFT freezes the original weights and learns a small parallel update, so the base knowledge is preserved and only lightly perturbed — less drift than full FT. PEFT reduces but does NOT eliminate forgetting.
  • Training loss decreases smoothly but inference output is garbage. Most likely bugs? Hard▸ reveal
    Smooth loss means optimization is fine — suspect serving/formatting: train/serve chat-template mismatch (most common), BOS/EOS or special-token differences, unmasked prompt tokens making loss misleadingly low, wrong generation config, or the wrong checkpoint/adapter loaded. Don't assume "the model didn't learn" — smooth train loss points to a formatting/serving mismatch.

  • SFT teaches behavior, not facts. (instruction → response) pairs + right template + loss masking (-100 on prompt).
  • LoRA is the default. ΔW = (α/r)BA, r(d+k) params; A random, B=0 so it starts at the base.
  • Merge = zero latency. Fold W₀ + (α/r)BA; or keep unmerged for hot-swap / multi-adapter serving.
  • QLoRA = 4-bit NF4 base + double quant + paged optimizers + bf16 compute → 70B on one 48GB GPU.
  • DoRA / rsLoRA / IA³ refine LoRA; PEFT resists catastrophic forgetting.
  • Choose by constraint: LoRA default · QLoRA when VRAM-bound · full FT for big shifts · RAG for new knowledge.
Recap the six pillars. If they remember nothing else: SFT changes behavior (mask the prompt), LoRA is the low-rank default that merges to zero latency, QLoRA is the 4-bit memory frontier, and you pick by your binding constraint — VRAM, data size, or whether you actually need new knowledge (then RAG).
