Pretraining & Scaling Laws

How a base model is trained from scratch on web-scale text — the self-supervised objectives that turn raw bytes into a next-token predictor, and the scaling laws that decide how big to make it and how long to train.

←→ navigate · S speaker notes · O overview · F fullscreen · append ?print-pdf to export

Welcome. This is module 06 — pretraining and scaling laws. Everything downstream (fine-tuning, RLHF, RAG, agents) starts from a base model, and this deck is about how that base model comes to exist. Two big threads: (1) the self-supervised objective — what game we make the model play over trillions of tokens, and (2) the scaling laws — the empirical rules that tell us how big to make the model (N parameters) and how many tokens (D) to train on for a given compute budget (C). By the end you should be able to derive C ≈ 6ND on a whiteboard, state Chinchilla's 20-tokens-per-parameter rule, and explain why GPT-3 was undertrained. Press S any time to read my script.

Roadmap

The core ideas to remember

  • The text labels itself — no humans in the loop.
  • Causal LM is the default — training = inference.
  • Loss is a power law in N, D, C — and it's predictable.
  • Scale N and D together (~20 tokens/param), not just N.
  • C ≈ 6ND is the budgeting workhorse.
  • Over-train for production — inference cost dominates lifetime cost.
These six bullets are the deck in miniature. Each one maps to a section. Tell the audience: by the time we close, every one of these should feel obvious. The subtle one is the last — Chinchilla minimizes training compute, but the thing you actually pay for over a model's life is inference, so the modern move is to deliberately over-train a smaller model.

Why it matters

Where pretraining sits in the LLM stack

Raw web text → Pretrainingthis module → Base model → Fine-tune / PEFT → RLHF / aligned chat
  • Pretraining is where the model learns the shape of language — grammar, facts, code, reasoning.
  • It's the most expensive stage — millions of GPU-hours, the bulk of the budget.
  • Everything after (SFT, RLHF) only steers a capability that pretraining already created.
Frame the economics: pretraining is the foundation everything else stands on, and it's where almost all the compute goes. Fine-tuning and RLHF are comparatively tiny — they reshape behavior but rarely add new raw knowledge. That's why "what data did it see in pretraining" determines so much. Mention that this is also the stage where the scaling-law decisions are irreversible: you commit to a parameter count and a data budget up front, and a single run can cost millions, so you cannot afford to guess the learning rate or the model size.

Why it matters

Why "self-supervised" changed everything

  • No labels needed — so we can use all of the internet, not a small annotated set.
  • The bottleneck moves from labeling to compute + data quality.
  • One simple game — "predict the hidden token" — scales to trillions of examples for free.
💡 Key shift Supervised learning needs humans to label every example. Self-supervision derives the label from the data itself, unlocking web-scale training.
This is the conceptual unlock. Before, the size of a labeled dataset capped how much a model could learn. Self-supervision removes that cap: every token in every document is a free training example. That's why the field pivoted from "collect more labels" to "collect more compute and cleaner data." It's worth saying explicitly that this is why LLMs got so big so fast — the supply of training signal became effectively unlimited.

Intuition

The flashlight analogy

Imagine reading every book in a giant library with a flashlight that covers the next word.

"The cat sat on the ___" → you guess mat, slide the light right, and check.

Wrong guesses nudge your mental model; right guesses reinforce it.

No teacher ever tells you the answer — the book already contains it.

After enough pages you've absorbed spelling, syntax, world facts, and reasoning.

That's exactly what a base LLM does — billions of times per second.

Tell this as a story; it's the most memorable framing in the whole deck. The point: there's no labeled dataset, just the corpus, and the "answer" is always the next word that's literally already written. Each prediction error is a learning signal. Played over trillions of tokens, this single game grows a system that can write essays and code. Emphasize that nothing about this is task-specific — grammar, facts, and reasoning all fall out of the same objective because predicting the next word well requires understanding all of those things.

The pretraining pipeline

The label for every token is just the next token already in the corpus — no humans in the loop.

Walk the pipeline left to right. Raw text (Common Crawl, code, books) → tokenize bytes into integer IDs → set up the self-supervised task (predict hidden token(s)) → score guesses against truth with cross-entropy → out pops a base model, which is just a next-token predictor. The crucial node is the third: the "task" is where all the different objectives we'll cover (causal, masked, span) plug in. Everything else in the pipeline is shared.

How it works

Causal (autoregressive) language modeling

Factorize a sequence left-to-right; predict each token from only the tokens before it:

$$ p_\theta(x) = \prod_{t} p_\theta(x_t \mid x_{<t}) $$

Used by every GPT-style decoder-only model — GPT-1/2/3, LLaMA, every modern chat base.

This is the dominant objective. The chain rule of probability lets us factor any sequence's probability into a product of next-token conditionals. The model's whole job is to estimate each p(x_t | x_<t). Three mechanics make this efficient and they're the next three slides: the causal mask, teacher forcing, and cross-entropy loss. Stress that "decoder-only" and "causal LM" go together — the architecture and the objective are matched.

How it works · mechanic 1

The triangular (causal) mask

  • Inside self-attention, add a mask of \(-\infty\) above the diagonal.
  • Position \(t\) can attend to positions \(\le t\) only.
  • If the future leaked in, the task is trivial — the model could just copy the answer.
  • And it would be useless at generation time, where the future doesn't exist yet.
The mask is what enforces "no peeking." Geometrically, the attention score matrix is n×n; we zero out (well, set to −∞ before softmax) everything above the diagonal so each position only sees itself and earlier positions. Why −∞? Because softmax(−∞) = 0, cleanly removing those connections. The deep reason: at inference we generate left to right and the future genuinely doesn't exist, so training must match that constraint. If we let position t see token t+1, the model learns to cheat and collapses at generation time.

How it works · mechanic 2

Teacher forcing

  • Feed the true tokens as input (not the model's own guesses).
  • Predict all n next-tokens in one parallel forward pass.
  • The label sequence is just the input shifted left by one.
  • → train on a whole document at once instead of one token at a time.
Teacher forcing is the trick that makes training fast. Instead of generating token by token and feeding back our own (possibly wrong) outputs, we feed the ground-truth sequence and ask the model to predict the shifted version. Because of the causal mask, every position's prediction only depends on earlier ground-truth tokens, so all n predictions are independent given the input and can be computed in a single forward pass. The labels are literally the inputs shifted left by one — predict x2 from x1, x3 from x1x2, and so on. This is why a single pass over a 4096-token document yields ~4096 training signals.

How it works · mechanic 3

Cross-entropy loss

  • At every position, compare the predicted distribution over the vocabulary to the one-hot true next token.
  • Sum the negative log-likelihood across positions.
  • Lower loss = better-calibrated next-token guesses.
📐 Reported as perplexity Perplexity \(= e^{\mathcal{L}/n}\) — the effective "branching factor" of the model's uncertainty.
Cross-entropy is just the negative log-probability the model assigned to the correct next token, summed over positions. A confident correct prediction contributes near-zero loss; a confident wrong one contributes a large loss. We typically average over tokens and report perplexity, e^(L/n), because it's interpretable: a perplexity of 10 roughly means the model is as uncertain as if it were choosing uniformly among 10 options at each step. Scaling laws are usually stated in terms of this loss.

How it works

What actually consumes the budget

  • Data curation — dedup, quality filtering, mixture weights → decides what the model reads.
  • Compute budget C ≈ 6ND + scaling laws → decides how big N and how many tokens D.
  • Optimizer schedule — warmup, cosine/WSD decay, μP → decides whether it converges or blows up.
The objective is conceptually simple; the hard, expensive engineering is everything around it. Three clusters: (1) data — what goes in determines what comes out; dedup and mixture weights are first-class decisions. (2) Sizing — the scaling laws and the C ≈ 6ND budget. (3) Dynamics — the learning-rate schedule, warmup, and μP, which decide whether your multi-million-dollar run converges smoothly or hits a catastrophic loss spike. We'll cover the objectives next, then the scaling cluster.

Approaches · 1 of 10

Causal / autoregressive LM the default

What: predict the next token given all previous ones. How: decoder-only Transformer + triangular mask + teacher forcing; loss = sum of per-token cross-entropies. Solves: generation is native — the training objective equals the inference task.

Pros
  • Generation = training; no head to bolt on
  • Every token = a loss term → dense signal
  • Scales predictably (Kaplan/Chinchilla fit it)
Cons
  • Strictly left-to-right (no future visibility)
  • Weaker for bidirectional encoding (NER, classification embeddings)

vs previous: the baseline — every later objective is defined relative to it.

Read more ↗

This is the baseline; everything else is defined relative to it. The killer property is that the training objective and the inference task are identical — you don't bolt on a classification head, you just sample. Dense signal: every single token gives a gradient. It also scales cleanly, which is why the scaling laws were all fit on causal LMs. The one real weakness is that context is strictly left-to-right, so for pure understanding tasks (named-entity recognition, sentence embeddings) a bidirectional encoder can beat it. Reference: GPT-3, Brown et al. 2020, arXiv 2005.14165.

How it works · Causal LM

Causal LM — step by step

x1..tcontext → causal mask≤ t only → logitsover vocab → −log p(xt+1)loss
  1. Feed the true tokens; the triangular mask lets position \( t \) see only positions \( \le t \) — no peeking at \( x_{>t} \).
  2. One parallel forward pass emits logits at every position (teacher forcing).
  3. Shift labels left by one; score each predicted distribution against the true next token.
  4. Sum cross-entropy \( \mathcal{L}=-\sum_t \log p_\theta(x_t \mid x_{<t}) \) — one pass over \( n \) tokens gives \( n \) signals.
Walk each step. The causal mask is what enforces "no peeking" — it sets attention scores above the diagonal to −∞ so softmax zeroes them. Teacher forcing feeds ground-truth tokens, so all n predictions are independent given the input and computed in a single pass. Labels are the input shifted left by one: predict x2 from x1, x3 from x1x2, and so on. That's why a single pass over a 4096-token document yields ~4096 training signals — the efficiency that makes web-scale pretraining feasible.

Approaches · 2 of 10

Masked LM (BERT) bidirectional

What: randomly mask ~15% of tokens, predict them from both sides (BERT, 2018). How: encoder with full (non-causal) attention; of masked positions, 80% → [MASK], 10% random, 10% unchanged (the 80/10/10 split). Solves: rich bidirectional representations for understanding (classification, NER, retrieval embeddings).

Pros
  • Bidirectional context → strong encoders
  • Great for discriminative / embedding tasks
Cons
  • Can't generate fluently (not autoregressive)
  • Only ~15% of tokens give signal → sample-inefficient
  • [MASK] never appears at inference (train/test mismatch)

vs previous: drops the causal mask entirely — attention is bidirectional, so each target sees left and right context. Great encoder, but it can't generate.

Read more ↗

How it differs from causal LM: BERT drops the causal mask entirely — attention is bidirectional, so each masked position is predicted from both left and right context. That makes it a fantastic encoder but unable to generate fluently. The 80/10/10 split is a classic interview detail: of the 15% chosen positions, 80% become [MASK], 10% become a random token, and 10% are left unchanged. Why? Because [MASK] never appears at fine-tune/inference time, so always using it creates a train/test mismatch; the random and unchanged cases force the model to build a good representation for every token, not just the masked slots. The big inefficiency: only ~15% of tokens contribute loss, versus 100% for causal LM. Reference: Devlin et al. 2018, arXiv 1810.04805.

How it works · Masked LM

Masked LM — step by step

sentencefull context → pick 15%80/10/10 → bi-dir encoderboth sides → predict maskedonly ~15%
  1. Sample ~15% of token positions as prediction targets.
  2. Corrupt them: 80% → [MASK], 10% → a random token, 10% left unchanged — so the model can't blindly trust the surface token.
  3. Encode with full attention; every position sees the whole sentence.
  4. Predict the original token at each target only — loss comes from ~15% of positions, not all of them.
Narrate the 80/10/10 rationale step by step: [MASK] never appears at inference, so always masking creates a train/test gap. The 10% random and 10% unchanged force a good representation for every token, and stop the model from assuming the observed token is always correct. Full bidirectional attention is the key contrast with causal LM — but it's exactly why BERT can't generate left-to-right, and why only the ~15% target positions yield a gradient, making it sample-inefficient per sequence.

Approaches · 3 of 10

Span corruption (T5) denoising

What: mask contiguous spans, reconstruct them as a target sequence (T5, 2019). How: each dropped span → a unique sentinel token (<X>, <Y>…) in the input; the decoder outputs sentinel-then-span pairs. Solves: an encoder-decoder objective that casts everything as text-to-text.

Pros
  • One text-to-text framing for all tasks
  • Shorter targets than full MLM → efficient
Cons
  • Encoder-decoder adds complexity vs decoder-only
  • Less natural for open-ended generation than causal LM

vs previous: masks whole spans (not single tokens) and predicts them seq2seq — shorter targets than MLM, and unifies every task as "text in, text out."

Read more ↗

How it differs from MLM: instead of masking individual tokens, T5 masks whole contiguous spans and replaces each with a single sentinel token (<X>, <Y>). The decoder then emits the sentinels paired with the original span text. This makes targets much shorter than MLM (you reconstruct only the dropped spans, not the whole sequence) and casts every NLP task into one text-to-text format — translation, summarization, classification all become "text in, text out." The cost is architectural: it's an encoder-decoder, which is heavier than decoder-only and less natural for open-ended generation. Reference: Raffel et al. 2019, arXiv 1910.10683.

How it works · Span corruption

Span corruption — step by step

documentthank you for inviting → drop spans→ <X> <Y> → encoder → decoder<X> span <Y> span
  1. Pick contiguous spans to corrupt; replace each with a single unique sentinel \( \langle X\rangle, \langle Y\rangle, \ldots \) in the input.
  2. The encoder reads the corrupted input bidirectionally.
  3. The decoder generates only the dropped spans, each preceded by its sentinel — the target is short, not the whole sequence.
  4. Every task (translate, summarize, classify) is framed the same way: text in → text out.
Step through the sentinel mechanism: a span like "for inviting me to your" collapses to one <X> token in the input, and the decoder's job is to emit "<X> for inviting me to your <Y> ...". This is the key efficiency win over MLM — targets are short because you only reconstruct what was dropped. And because input and output are both text, T5 casts translation, summarization, and classification into one uniform format. The trade is the encoder-decoder overhead versus a plain decoder-only stack.

Approaches · 4 of 10

Prefix-LM & UL2 "mixture-of-denoisers" unified

What: Prefix-LM — a prefix attends bidirectionally, the continuation is causal. UL2 (2022) trains a mixture-of-denoisers. How: UL2 mixes R (regular short spans, T5-like), X (extreme/long-span), S (sequential / prefix-LM) with a mode token saying which game. Solves: one model good at both understanding and generation.

Pros
  • One model spans MLM + span + causal strengths
  • Mode tokens pick behavior at inference
Cons
  • More moving parts; mixing ratios are extra hyperparameters
  • Pure decoder-only causal LM won most production stacks

vs previous: instead of one denoiser (T5), it trains a mixture and signals the mode — bidirectional understanding plus causal generation in one model.

Read more ↗

How it differs from T5: rather than commit to one denoiser, UL2 trains on a mixture and tells the model which mode it's in via a special mode token. R-denoising is short spans (like T5), X-denoising is aggressive long-span corruption, and S-denoising is sequential — essentially prefix-LM / causal generation. Prefix-LM itself is the bridge: the prefix gets bidirectional attention (encoder-like), the continuation is generated causally (decoder-like) in a single stack. The promise is "one model, both skills." The reality, as the cons note, is that the field largely converged on pure decoder-only causal LMs for production because they're simpler and scale just as well. Reference: Tay et al. 2022, arXiv 2205.05131.

How it works · Prefix-LM / UL2

Prefix-LM / UL2 — step by step

mode token[R] [X] [S] → prefixbi-directional → targetcausal → one modelunderstand + generate
  1. Prefix-LM: the prefix tokens attend to each other freely (encoder-like); generated tokens attend to the full prefix plus prior targets (causal).
  2. UL2 prepends a mode token — R (short spans), X (long/extreme spans), S (sequential, i.e. prefix-LM).
  3. Each batch samples a denoiser; the model learns all three games at once.
  4. At inference, pick the mode token to switch between understanding and generation behavior.
Walk the attention pattern: in prefix-LM, only the prefix is bidirectional, so it's a single decoder stack with a modified mask — encoder-like conditioning on the input, autoregressive generation of the output. UL2 generalizes this: R/X/S denoisers sit on a continuum of span length and masking rate, and a mode token tells the model which one. The headline is "one model, both skills" — but in practice pure decoder-only causal LMs won production because they're simpler and scale just as well. The mixing ratios become extra hyperparameters you have to tune.

Approaches · 5 of 10

Fill-in-the-Middle (FIM) code

What: teach a causal LM to fill a gap given both a prefix and a suffix (Bavarian et al., 2022) — essential for IDE code completion. How: split a doc into prefix / middle / suffix, reorder to <PRE> prefix <SUF> suffix <MID> middle (the PSM layout). Free: still plain next-token training — adding FIM doesn't hurt left-to-right ability.

Pros
  • Enables insertion/infilling (IDE autocomplete)
  • "Free": same AR loss, no left-to-right regression
Cons
  • Needs careful sentinel/format handling
  • Mix rate (e.g. 50–90% FIM) is a tuning knob

vs previous: a pure data transform on causal LM — reorder so the suffix precedes the middle, and the same next-token loss learns to infill.

Read more ↗

How it differs from plain causal LM: a left-to-right model can only continue text to the right — it can't fill a gap in the middle of a file, which is exactly what you need when editing code in an IDE. FIM's trick is purely a data transformation: take a document, cut it into prefix / middle / suffix, and reorder it so the suffix comes before the middle, marked with sentinels (<PRE> <SUF> <MID>). Now the same vanilla next-token objective learns to produce the middle given both sides. The "FIM-for-free" headline result is that mixing in FIM at training time doesn't degrade ordinary left-to-right performance, so you get infilling at no quality cost. The knobs are the format (PSM vs SPM ordering) and the FIM mix rate, typically 50–90%. Reference: Bavarian et al. 2022, arXiv 2207.14255.

How it works · FIM

FIM — step by step

docprefix · middle · suffix → reorder<PRE><SUF><MID> → causal LMsame AR loss → predict middlegiven both sides
  1. Take a document and split it into prefix / middle / suffix.
  2. Reorder to <PRE> prefix <SUF> suffix <MID> middle so the suffix appears before the middle.
  3. Train with the ordinary causal next-token loss — nothing about the objective changes.
  4. Apply to a fraction of docs (≈ 50–90%); left-to-right ability is preserved, so infilling is "for free."
The whole trick is data reordering. By moving the suffix in front of the middle, the same left-to-right model now conditions on both sides when it reaches the <MID> sentinel — it learns to infill. The "FIM-for-free" result is the headline: mixing it in at ~50–90% doesn't degrade ordinary completion. The two knobs are the layout (PSM vs SPM ordering) and the mix rate. This is what powers IDE autocomplete, where you edit inside a file rather than at the end.

Approaches · 6 of 10

Kaplan scaling laws 2020

What: first big empirical study — test loss falls as a smooth power law in N, D, C. How: a straight line on a log-log loss-vs-compute plot. Recommended: with more compute, make models much bigger, increase data only modestly. Caveat: that N-vs-D advice was later shown biased (LR schedule + embedding accounting).

Pros
  • First to show clean, predictable power laws
  • Justified ever-larger models (→ GPT-3)
Cons
  • Over-weighted parameters → too little data
  • Fixed-length cosine + warmup skewed the exponents

vs previous: shifts from "what game" (objectives) to "how much compute" — and made loss predictable before training.

Read more ↗

Kaplan et al. 2020 (arXiv 2001.08361) was the breakthrough that made loss predictable: you could fit a power law from small runs and forecast a huge model's loss before training it. That predictability is why people felt safe spending GPT-3 money. But the methodology had a flaw — they used a fixed-length cosine schedule and counted embedding parameters in a way that biased the exponents, leading to the conclusion "spend compute mostly on parameters." That advice produced GPT-3, which we'll see was badly undertrained. This sets up the Chinchilla correction.

How it works · Kaplan

Kaplan — step by step

  1. Train many small models, varying params \( N \), data \( D \), compute \( C \).
  2. Plot test loss on log-log axes — the points fall on a straight line: \( L \propto C^{-\alpha} \).
  3. Extrapolate that line to forecast a giant model's loss before spending the money.
  4. Their (later-corrected) advice: with more compute, grow \( N \) faster than \( D \).
The power-law insight: each multiplicative increase in compute yields a constant fractional drop in reducible loss, so a straight log-log line lets you extrapolate. That predictability is the real contribution. The flaw is in step 4: a fixed-length cosine schedule (not matched to each run's horizon) plus counting embedding params biased the exponents toward "make it bigger." That recommendation produced GPT-3 — and Chinchilla showed it was wrong.

Approaches · 7 of 10

Chinchilla compute-optimal ~20 tokens/param

What: the corrected result (Hoffmann et al., 2022) — for fixed compute, grow N and D in equal proportion → ~20 tokens/param. How: swept hundreds of models with properly tuned LR schedules; fit \( L(N,D)=E+A/N^\alpha+B/D^\beta \). Result: Chinchilla (70B, 1.4T) beat Gopher (280B) with 4× fewer params.

Pros
  • Smaller, cheaper-to-serve models match bigger ones
  • Clean parametric loss law to plan budgets
Cons
  • "Optimal" ignores inference cost (training-only)
  • Assumes abundant unique data — not always true

vs previous: same power-law philosophy, fixed methodology — matched the LR schedule to each run, so N and D scale almost equally, not parameters-first.

Read more ↗

How it differs from Kaplan: same power-law philosophy, fixed methodology. By tuning the LR schedule to the run length (not a fixed cosine), Hoffmann et al. found that N and D should scale almost equally — the exponents α≈0.34, β≈0.28 are close enough that the optimal ratio is roughly 20 tokens per parameter. The dramatic proof: Chinchilla, a 70B model on 1.4T tokens, beat the 280B Gopher despite being 4× smaller, just by being trained on the right amount of data. The headline lesson: Kaplan-era models were undertrained — too big for their data budget. Caveats: "optimal" here means training-compute-optimal and ignores inference cost, and it assumes you have enough unique data. Reference: arXiv 2203.15556.

How it works · Chinchilla

Chinchilla — step by step

  1. For a fixed compute budget (\( C \approx 6ND \)), sweep model size \( N \) — loss is U-shaped.
  2. The bottom of each valley is the best \( N \) for that budget — too small underfits, too big starves on tokens.
  3. Connect the minima → the compute-optimal frontier: \( N_{\text{opt}}, D_{\text{opt}} \propto C^{0.5} \).
  4. Along it \( D/N \approx 20 \) — scale params and tokens together.
Use the interactive widget: hit Play/Step. With 10× compute you grow both N and D by √10 ≈ 3.2×, keeping D/N near 20. The fix versus Kaplan was matching the LR schedule to each run's horizon so shorter runs weren't penalized. α≈0.34 and β≈0.28 are close enough that the optimum scales the two roughly equally. The proof point: 70B Chinchilla on 1.4T tokens beat 280B Gopher. If someone asks "why 20 tokens/param," it falls out of these exponents. Caveat: this is training-optimal and ignores inference — which motivates over-training next.

Approaches · 8 of 10

Over-training / inference-optimal LLaMA

What: deliberately train a smaller model on far more tokens than Chinchilla-optimal (LLaMA, 2023 — e.g. 7B on 1T+ tokens, ≫ 20×). Why: Chinchilla minimizes training compute; a deployed model pays inference cost on every request, forever. Solves: total lifetime cost (train + serve) — what actually matters in production.

Pros
  • Cheaper, faster inference at equal quality
  • Smaller weights fit edge / single-GPU deployment
Cons
  • More training compute than Chinchilla-optimal
  • Diminishing returns — loss keeps falling but slowly

vs previous: Chinchilla minimizes training FLOPs; over-training trades extra training for a permanently cheaper-to-serve model.

Read more ↗

How it differs from Chinchilla: Chinchilla answers "minimize training FLOPs." But once you deploy, you pay inference cost on every single request for the model's entire life — and that usually dwarfs training. So if a smaller-but-over-trained model can hit the same quality, it's cheaper overall because each inference is cheaper. LLaMA-7B trained on 1T+ tokens is way past the ~140B tokens Chinchilla would prescribe for 7B, and that's intentional. The trade is: you spend more training compute for a model that's permanently cheaper to serve and small enough to fit on one GPU. Returns diminish — loss keeps dropping but slowly — so there's a sweet spot. Reference: Touvron et al. 2023, arXiv 2302.13971.

How it works · Over-training

Over-training — step by step

  1. Pick a deliberately small \( N \) for cheap serving (e.g. 7B); loss falls fast at first.
  2. Stop at the Chinchilla optimum (≈20:1, ~140B tokens for 7B) → minimized training compute.
  3. Keep going — LLaMA-7B saw 1T+ tokens. The loss drop is tiny, but the model stays small.
  4. Win at deployment: every request runs the small model, amortizing the extra training over billions of calls.
The economics drive every step: training is paid once, inference is paid per request forever, and inference cost scales with parameter count. So pick a small N, then over-train to recover quality. LLaMA-7B on 1T+ tokens is the canonical example — well past Chinchilla-optimal on purpose. Returns diminish (the loss curve flattens), so there's a sweet spot rather than "more is always better." The model also fits on a single GPU / edge device, which Chinchilla-optimal sizing might not.

Mixture-of-Experts decouples total parameters from per-token compute. You have many expert feed-forward networks, but a gating network routes each token to only the top-k (often k=1 or 2), so a model can have 100B+ total parameters while activating only ~10–20B per token. Scaling laws then depend on active parameters and the number of experts, not raw total. The win is capacity-per-FLOP: more memorized knowledge without proportional compute. The costs are real though — all experts must sit in memory (huge footprint), routing can be unstable so you need load-balancing losses, and serving is more complex. Reference: Hugging Face, "Mixture of Experts Explained," huggingface.co/blog/moe.

How it works · MoE

MoE — step by step

token → gating netscores experts → top-k expertsk = 1 or 2 → combine~10–20B active
  1. A token enters a layer with many expert feed-forward networks instead of one.
  2. A small gating network scores the experts and selects the top-\( k \) (often \( k=1 \) or \( 2 \)).
  3. Only those experts compute — so total params can be 100B+ while only ~10–20B are active per token.
  4. Combine the selected experts' outputs by gate weights; add a load-balancing loss so routing doesn't collapse onto a few experts.
Use the widget to show routing: the gate scores all experts, top-k fire, the rest stay idle, outputs are combined. The headline is decoupling total params from per-token FLOPs — capacity for memorized knowledge without proportional compute. Scaling laws gain a sparsity axis: loss depends on active params and the number of experts, and there's an optimal sparsity. The costs are concrete — all experts must sit in memory (big footprint), routing can be unstable so you add load-balancing (auxiliary) losses, and distributed serving has communication overhead.

The math

Causal-LM cross-entropy loss

$$ \mathcal{L}(\theta) = -\sum_{t=1}^{n} \log p_\theta(x_t \mid x_{<t}) $$

\(\mathcal{L}(\theta)\) = loss for one sequence; \(\theta\) = model weights; \(n\) = sequence length.

\(x_t\) = token at position \(t\); \(x_{<t}\) = all tokens before \(t\); \(p_\theta\) = predicted probability of the true next token.

Negative log → confident-correct ≈ 0 loss, confident-wrong → large. Average over tokens; report perplexity \(= e^{\mathcal{L}/n}\).

Walk it term by term as the fragments appear. The sum runs over every position; at each one we take the negative log-probability the model assigned to the actual next token. The shape of −log: it punishes confident mistakes harshly and rewards confident correct guesses with near-zero loss. In practice we average over tokens so the number is comparable across sequence lengths, and we exponentiate to get perplexity, which is the intuitive "branching factor" of the model's uncertainty.

The math

Chinchilla parametric loss law

$$ L(N,D) = E + \frac{A}{N^{\alpha}} + \frac{B}{D^{\beta}} $$

\(L\) = expected test loss; \(N\) = (non-embedding) parameters; \(D\) = training tokens.

\(E\) = irreducible loss — the entropy of language you can never beat, even with infinite data & params.

\(A/N^{\alpha}\) = error from a finite model (shrinks as you add params); \(B/D^{\beta}\) = error from finite data (shrinks as you add tokens).

Fit \(\alpha \approx 0.34,\ \beta \approx 0.28\) — close enough that \(N\) and \(D\) should scale almost equally.

This is the single most important equation in the deck. Three pieces: E is the floor — the inherent unpredictability of language that no model can beat. The A/N^α term is your "model is too small" error and it shrinks as you add parameters. The B/D^β term is your "not enough data" error and it shrinks as you add tokens. Because α and β are both around 0.3 and roughly equal, the two error terms shrink at similar rates — which is precisely why the optimum scales N and D together rather than favoring one. If someone asks "why 20 tokens per param," it falls out of these exponents.

The math

Compute estimate

$$ C \approx 6ND \;\;\text{FLOPs} $$

\(C\) = total training compute (FLOPs); \(N\) = parameters; \(D\) = tokens.

The 6 counts work per parameter per token: forward ≈ 2 FLOPs/param (one multiply + one add), backward ≈ 4 (grads w.r.t. inputs and weights) → 2 + 4 = 6.

Back-of-envelope: ignores attention's quadratic term, which only matters at very long sequences.

This is the budgeting workhorse and a guaranteed interview question. The factor 6 is not magic — it's 2 for the forward pass (a matmul does one multiply and one add per parameter per token) and 4 for the backward pass (you compute gradients with respect to both the activations/inputs and the weights, roughly doubling the forward cost). 2 + 4 = 6. Multiply by N parameters and D tokens and you get total FLOPs. It's an approximation that drops the attention O(n²) term, which is negligible unless sequences are very long.

The math

The Chinchilla optimum

$$ N_{\text{opt}} \propto C^{\,a}, \quad D_{\text{opt}} \propto C^{\,b}, \quad a \approx b \approx 0.5 \;\Rightarrow\; \frac{D}{N} \approx 20 $$

\(N_{\text{opt}}, D_{\text{opt}}\) = loss-minimizing param & token counts for a fixed budget \(C\).

Both exponents ≈ 0.5 → doubling compute means scaling both \(N\) and \(D\) by ≈ \(\sqrt{2}\) (1.41×), holding \(D/N \approx 20\).

Practically: a 10B-param model wants ≈ 200B tokens to be compute-optimal.

This is the operational takeaway from the loss law. Because both exponents are near 0.5, parameters and tokens should grow at the same rate as compute increases — doubling your budget means making the model about 1.41× bigger AND training on 1.41× more tokens, which keeps the ratio pinned near 20:1. The concrete rule to memorize: tokens ≈ 20 × parameters. A 10B model wants ~200B tokens; a 70B model wants ~1.4T tokens (which is exactly Chinchilla). We'll use this in the worked example next.

In code

Causal-LM loss with teacher forcing

import torch
import torch.nn.functional as F

def causal_lm_loss(logits, labels, pad_id=-100):
    # logits: (batch, seq, vocab)  labels: (batch, seq) of token ids
    # Predict token t+1 from position t -> drop last logit, drop first label.
    shift_logits = logits[:, :-1, :].contiguous()   # (b, n-1, vocab)
    shift_labels = labels[:, 1:].contiguous()        # (b, n-1) shifted by one
    loss = F.cross_entropy(
        shift_logits.view(-1, shift_logits.size(-1)),  # (b*(n-1), vocab)
        shift_labels.view(-1),                          # (b*(n-1),)
        ignore_index=pad_id,   # padding positions contribute no loss
    )
    return loss   # mean negative log-likelihood per (non-pad) token
This is the entire causal-LM objective in ~10 lines. The two load-bearing lines are the shifts: drop the last logit (it predicts a token past the end) and drop the first label (nothing predicts the very first token), so position t's logits line up against token t+1. Then flatten to (b·(n−1), vocab) and call cross_entropy. ignore_index=pad_id means padded positions don't contribute loss. Point out that this is literally what every HuggingFace causal model does under the hood.

In code

Back-of-envelope scaling helper

import math

def training_flops(N, D):
    "FLOPs to train N params on D tokens (C = 6*N*D)."
    return 6 * N * D

def chinchilla_optimal(C, tokens_per_param=20):
    "Given a compute budget C, return compute-optimal (N, D)."
    # C = 6*N*D and D = r*N  ->  C = 6*r*N^2  ->  N = sqrt(C / (6*r))
    r = tokens_per_param
    N = math.sqrt(C / (6 * r))
    D = r * N
    return N, D

C = training_flops(70e9, 1.4e12)         # 70B params, 1.4T tokens
print(f"{C:.2e} FLOPs")                   # ~5.9e23
N, D = chinchilla_optimal(C)
print(f"optimal N={N/1e9:.0f}B  D={D/1e12:.2f}T")  # ~ 70B / 1.4T
Two helpers. training_flops just applies C = 6ND. chinchilla_optimal inverts it: fix the ratio r = D/N = 20, substitute D = rN into C = 6ND to get C = 6rN², solve N = sqrt(C / 6r), then D = rN. Run it on a 70B/1.4T budget and it returns ~70B / 1.4T — confirming Chinchilla sits exactly at its own optimum. This is the calculator you'd actually use to plan a run.

Worked example

Sizing a 70B run — the FLOPs

How many FLOPs to train a 70B model on 1.4T tokens? Plug into C ≈ 6ND:

$$ C \approx 6 \times (70\times10^{9}) \times (1.4\times10^{12}) \approx 5.9\times10^{23}\ \text{FLOPs} $$

≈ 590 zettaFLOPs. On a cluster sustaining \(10^{18}\) useful FLOP/s (after MFU losses): \(5.9\times10^{23}/10^{18} \approx 5.9\times10^{5}\) s ≈ ~7 days wall-clock.

Do the arithmetic live. 6 × 70e9 × 1.4e12: first 70e9 × 1.4e12 = 9.8e22, times 6 ≈ 5.9e23 FLOPs. That's ~590 zettaFLOPs. To turn FLOPs into time, divide by useful throughput — say an exaFLOP (1e18 useful FLOP/s) after accounting for Model FLOPs Utilization (you never get peak). That gives ~5.9e5 seconds, about a week. This is exactly how you'd sanity-check a training plan on a napkin.

Worked example

Is 1.4T optimal for 70B?

Chinchilla says \(D \approx 20N\):

$$ D_{\text{opt}} \approx 20 \times 70\times10^{9} = 1.4\times10^{12} = 1.4\text{T tokens} \;\;\checkmark $$

So 70B-on-1.4T is almost exactly Chinchilla-optimal — which is precisely why Chinchilla was trained at that point.

Apply the 20:1 rule directly: 20 × 70B = 1.4T. The model and its data budget line up perfectly, confirming that the original Chinchilla paper trained its namesake model right at the compute-optimal frontier it had just discovered. This is a clean, satisfying check that ties the loss law to a real model.

Worked example

Contrast: GPT-3 was undertrained

  • GPT-3: \(N = 175\text{B}\) but only \(D = 300\text{B}\) tokens → ratio ≈ 1.7 tokens/param.
  • That's about 12× below the Chinchilla target.

$$ D_{\text{opt}} \approx 20 \times 175\times10^{9} = 3.5\times10^{12} = 3.5\text{T tokens} $$

GPT-3 spent its budget on raw parameters instead of data. The headline lesson: for the same compute, a smaller model trained on more tokens often wins.

This is the punchline of the whole scaling story. GPT-3 had 175B parameters but saw only ~300B tokens — a ratio of 1.7, about 12× short of the 20:1 target. To be compute-optimal it should have seen ~3.5T tokens. So GPT-3 "wasted" much of its budget on parameters that weren't fed enough data. A Chinchilla-style allocation of the same compute would have produced a smaller model trained on far more tokens, with lower loss and cheaper inference. This single example is why Chinchilla changed how the whole field allocates compute.

Decision guide

Walk the tree. First fork: do you need generation at all? If not — you want embeddings, classification, NER — reach for a masked-LM encoder like BERT/RoBERTa. If yes, are you building a code model? When to use what

The practical checklist

  • Generative LLM? Causal/decoder-only — training = inference.
  • Embeddings / classification? Masked-LM encoder (BERT/RoBERTa).
  • Code model? Add FIM to causal (≈ 50–90% mix).
  • Budget planning? Start at Chinchilla ~20:1, then over-train if you'll serve at scale.
  • Out of unique data? Repeat up to ~4 epochs, then add params over more repetition.
  • Max capacity per FLOP? MoE — if you can afford the memory + routing complexity.
  • Tuning a giant run? μP to transfer LR from small models; WSD to fork checkpoints.
This is the field guide. Each line maps to an approach we covered. Stress the two production-level instincts: (1) over-train when inference dominates, and (2) use μP so you never guess the learning rate on a run that costs millions. The 4-epoch rule is the practical answer to the "running out of data" worry. WSD (warmup-stable-decay) is worth mentioning because its long constant-LR phase lets you checkpoint and fork, which is great for scaling-law sweeps.

Pitfalls & misconceptions

People think X, but actually Y (1/2)

⚠ "Bigger is always better." A model can be too big for its data — GPT-3 was undertrained. For fixed compute, more tokens on a smaller model often wins.
⚠ "Chinchilla = exactly 20 tokens/param, always." 20:1 minimizes training compute only. If inference dominates lifetime cost, over-training (LLaMA-style) is better.
⚠ "The 6 in 6ND is arbitrary." It's 2 (forward) + 4 (backward) FLOPs/param/token — derivable, not magic.
⚠ "MLM and causal LM are interchangeable." MLM is bidirectional and can't generate; causal LM is autoregressive and generates natively. Different downstream tasks.
These are the misconceptions interviewers love to surface. "Bigger is always better" — no, GPT-3 proved you can be too big for your data. "20:1 always" — no, that's training-optimal; production over-trains. "6 is arbitrary" — no, 2+4, derive it. "MLM ≈ causal LM" — no, fundamentally different: one's an encoder that can't generate, the other generates natively. For each, state the wrong belief, then the correction, crisply.

Pitfalls & misconceptions

People think X, but actually Y (2/2)

⚠ "Duplicate data is harmless, just redundant." Dedup matters a lot — duplicates waste compute, encourage memorization/regurgitation, and can leak test data. Near-dedup (MinHash + LSH) is standard.
⚠ "Emergence proves a magical phase transition." Some "emergence" is a metric artifact — discontinuous metrics (exact-match) make smooth gains look sudden.
⚠ "A high benchmark score means a good model." Might be data contamination — the test set leaked into pretraining. Always decontaminate.
⚠ "Continued pretraining is free capability." It risks catastrophic forgetting — without replay or LR re-warming, adapting can wreck general ability.
Second batch. Dedup is underrated: duplicates burn compute, drive verbatim memorization (a privacy and copyright problem), and can leak eval data into training. Near-dedup with MinHash + LSH is the standard tool. The emergence "mirage" is a live debate — Schaeffer et al. showed that switching from a discontinuous metric (exact match) to a smooth one often turns an apparent sudden jump into gradual improvement. Contamination: a sky-high benchmark might just mean the test leaked into pretraining, so decontaminate and cross-check related benchmarks. And continued pretraining can cause catastrophic forgetting unless you replay old data or re-warm the learning rate.

What's new

From "make it bigger" to "feed it smarter"

🚀 The modern shift
  • Over-training is standard. Frontier teams train at 100–1000+ tokens/param (≫ 20) — inference cost dominates.
  • Data quality > quantity. Aggressive dedup, classifier-based filtering, learned mixture weights (DoReMi).
  • Hyperparameter transfer via μP. Tune LR on a tiny model, transfer to a huge one.
  • WSD / trapezoidal schedules. Warmup-Stable-Decay (MiniCPM) — checkpoint & fork from the stable phase.
  • The "emergence" debate. Wei et al. (sudden) vs Schaeffer et al. (mirage from discontinuous metrics).
The center of gravity moved from "scale parameters" to "scale data quality and improve dynamics." Over-training at 100–1000+ tokens/param is now routine because deployment cost dominates. DoReMi uses a small proxy model to learn the optimal domain mixture weights rather than guessing them. μP (maximal-update parametrization) is the practical breakthrough for big runs — you tune the LR cheaply on a tiny model and transfer it, eliminating the scariest guess in a multi-million-dollar run. WSD replaces fixed-length cosine with a long constant-LR phase you can checkpoint and fork, plus a short decay — ideal for scaling-law sweeps. And the emergence debate is genuinely unresolved methodology, worth flagging as an open question.

Full bank: 06 · Pretraining & Scaling Laws — from the answer key, condensed.

  • What is the causal (autoregressive) LM objective, and what does the model predict at each position? Easy▸ reveal
    It factorizes the joint as a product of conditionals \( P(x)=\prod_t P(x_t \mid x_{<t}) \): at each position predict the next token given only preceding tokens, trained with cross-entropy. One forward pass scores every position at once.
  • Explain MLM as used in BERT. What fraction of tokens are masked and why? Easy▸ reveal
    Randomly select ~15% of tokens and reconstruct them from bidirectional context. 15% balances signal vs efficiency — too few gives weak signal, too many destroys the context you predict from. Strong for understanding/encoding, not generation.
  • Why does causal LM use a triangular mask, and what breaks if future tokens leak? Easy▸ reveal
    A lower-triangular mask makes position \( t \) attend only to positions \( \le t \), enforcing the autoregressive factorization. If the future leaks, the target is visible in the input — the model "sees the answer," loss collapses, and it's broken at inference. It's about preventing label leakage, not efficiency.
  • Describe T5's span corruption. How does it differ from token-level MLM? Med▸ reveal
    T5 masks contiguous spans, replacing each with one unique sentinel; the decoder generates the dropped spans seq2seq-style, each preceded by its sentinel. Unlike BERT's in-place single-token MLM, it predicts variable-length multi-token spans, giving shorter targets and matching generative use.
  • Why are decoder-only autoregressive models dominant despite MLM being more bidirectional? Med▸ reveal
    Generation is inherently sequential, which causal LM does natively. It gives a signal at every token (vs ~15% for MLM) so it's more sample-efficient, scales cleanly, and enables zero/few-shot prompting. Bidirectionality mainly helps representation tasks, which matter less for general assistants.
  • What is fill-in-the-middle (FIM), why is it valuable for code, and how is the data reformatted? Med▸ reveal
    FIM teaches a causal model to infill a span given both prefix and suffix — essential when editing inside code files. Split a doc into prefix/middle/suffix and reorder to [PRE]prefix[SUF]suffix[MID]middle, so standard next-token training learns to condition on the suffix. Applied to ~50% of docs, it adds infilling with negligible left-to-right loss.
  • Why the 80/10/10 [MASK]/random/original split in BERT? Hard▸ reveal
    [MASK] never appears at fine-tune/inference, creating a train–test mismatch. Of the ~15% targets, 80% become [MASK], 10% random, 10% unchanged. This forces a good representation for every position and stops the model blindly trusting the surface token. Reduces the pretrain/fine-tune distribution gap.
  • According to Chinchilla, what is the approximate optimal ratio of training tokens to parameters? Easy▸ reveal
    Roughly 20 tokens per parameter — scale \( N \) and \( D \) in equal proportion, so a 10B model wants ~200B tokens. Kaplan's earlier work implied far fewer tokens/param (favoring bigger models); Chinchilla corrected it.
  • Why was GPT-3 (175B params, ~300B tokens) considered "undertrained"? Med▸ reveal
    That's ~1.7 tokens/param, ~12× below the ~20 optimum. Under the same compute a much smaller model on far more tokens would reach lower loss. GPT-3 spent its budget on parameters rather than data, leaving performance on the table.
  • With 10× more compute than a Chinchilla-optimal model, how do you scale \( N \) and \( D \)? Med▸ reveal
    Since \( C \approx 6ND \) and both \( N \) and \( D \) scale as ~\( C^{0.5} \), increase each by \( \sqrt{10}\approx 3.16\times \). The ~20 tokens/param ratio is preserved. Don't multiply only parameters (or only tokens) by 10.
  • Where does the factor of 6 in \( C\approx 6ND \) come from? Hard▸ reveal
    Each weight is one multiply-accumulate (~2 FLOPs) per token in the forward pass; the backward pass is ~2× that (~4 FLOPs) because it computes gradients w.r.t. both activations and weights. So ~2 + 4 = 6 FLOPs per param per token. Attention score/softmax costs are neglected.
  • Write the Chinchilla loss \( L(N,D)=E+A/N^\alpha+B/D^\beta \) and explain each term. Hard▸ reveal
    \( E \) = irreducible loss (data entropy / Bayes floor). \( A/N^\alpha \) = reducible error from finite model capacity (vanishes as \( N\to\infty \)). \( B/D^\beta \) = reducible error from finite data (vanishes as \( D\to\infty \)). α and β are close but NOT equal (≈0.34 vs ≈0.28).
  • How do scaling laws change for sparse MoE, and what does decoupling params from per-token FLOPs buy? Hard▸ reveal
    MoE routes each token to a few experts, so total params grow without raising per-token FLOPs. Laws gain a sparsity / expert-count axis, and for fixed FLOPs MoE typically reaches lower loss than a dense model. The win is memory-for-compute. Cost: bigger memory footprint, routing complexity, and communication overhead.
  • Given 1024 A100s at known achieved FLOP/s, estimate wall-clock for a FLOP budget. Hard▸ reveal
    Time = total FLOPs ÷ (GPUs × achieved FLOP/s). E.g. A100 ~312 TFLOP/s bf16, at ~40% MFU ≈ 125 TFLOP/s → cluster ≈ 1.28×10¹⁷ FLOP/s. A 6×10²³-FLOP budget ≈ 54 days. Use achieved (MFU-adjusted), not peak — peak overestimates speed 2–3×.
  • Train loss falls smoothly but a downstream metric stays flat then jumps — how do you interpret it? Med▸ reveal
    Typical of a thresholded metric: the model steadily improves (log-likelihood of the right answer rising) but the metric only registers once probability mass crosses the decision threshold, looking like sudden "emergence." Don't conclude a new ability emerged — it's usually a metric artifact.

Takeaways

What to remember

  • Pretraining is self-supervised — the next token is its own label; learn from unlabeled web-scale text.
  • Causal/decoder-only LM is the default (training = inference); MLM, span, UL2, FIM are alternatives for different tasks.
  • Scaling laws are power laws — L(N,D)=E+A/Nᵅ+B/Dᵝ; loss falls predictably with N, D, C.
  • Chinchilla's ~20 tokens/param revised Kaplan — scale N and D together; GPT-3 was badly undertrained.
  • Compute ≈ 6ND (2 fwd + 4 bwd); MFU tells you how much peak hardware you actually use.
  • Over-training pays off in production — when inference dominates, train smaller on far more tokens.
Recap by walking each bullet — these mirror the roadmap, closing the loop. If the audience leaves with these six, they can hold their own on any pretraining/scaling question. The deepest practical insight is the last two together: 6ND lets you budget, and the over-training instinct corrects naive Chinchilla for real-world inference cost.

Takeaways · the frontier

Where the field is going

  • Data and dynamics are the new frontier: dedup + learned mixtures (DoReMi), μP transfer, WSD schedules.
  • Loss-spike control, the emergence "mirage" debate, contamination, and continued pretraining without forgetting.

Go deeper: Chinchilla (Hoffmann et al., 2022) · Harm de Vries — "Go smol or go home" (over-training math).

The future isn't "make it bigger" — that lever is mostly pulled. It's smarter data (dedup, learned mixtures), better dynamics (μP, WSD), and a cluster of open problems: loss spikes, the emergence debate, contamination, and continued pretraining without catastrophic forgetting.
