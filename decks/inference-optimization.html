Inference Optimization & Serving

Where cost and latency are actually won or lost: quantization, the KV-cache, PagedAttention & vLLM, continuous batching, parallelism, and speculative decoding — and the latency/throughput/cost trade-offs of running models in production.

Welcome. Training gets the headlines, but serving is where the bill lands — every token a user reads costs GPU-seconds, and every millisecond of delay is felt. This module is about squeezing a trained model onto hardware and serving it fast and cheap without wrecking quality. The single sentence the whole lecture turns on: decode is memory-bandwidth bound, not compute bound — once you internalize that, batching, the KV-cache, quantization, and speculative decoding all click into place. Core ideas to land

Five things to take away

  • Decode is memory-bandwidth bound, not compute bound — you're waiting on weights, not on FLOPs.
  • The KV cache is the central object — size it; it, not the weights, sets concurrency.
  • Batching trades latency for throughput — more requests per pass, slightly slower per request.
  • PagedAttention = virtual memory for the KV cache — pages kill fragmentation.
  • Measure goodput under an SLO, not raw tok/s — useful throughput is the only honest number.
This is the executive summary — I come back to it as the recap. Notice how they chain. Decode being bandwidth-bound is why batching works (you read the weights once and amortize them over many requests) and why quantization helps (smaller weights = less to read). The KV cache is what fills the GPU and caps concurrency, so paging it (PagedAttention) is what lets you batch more. And goodput is the scoreboard: a config that posts huge tok/s while blowing the latency SLO is serving zero useful requests. Those five sentences are the spine of the whole deck.

Why it matters

Serving is where the money & the latency live

  • Training is a one-time capital cost; serving is a perpetual per-token cost.
  • Every generated token burns GPU-seconds; every extra millisecond is felt by the user.
  • The same model can be 10× cheaper or faster depending only on how you serve it.
The framing for the whole module: you pay to train a model once, but you pay to serve it on every single request, forever. At scale the serving bill dwarfs the training bill. And the leverage is enormous — the same weights, served naively versus served well, differ by an order of magnitude in both cost-per-token and tail latency. That gap is entirely engineering: quantization, the KV-cache layout, batching policy, and parallelism. So this is not a niche systems topic; it's where the economics of an LLM product are decided.

Why it matters · two phases

Prefill vs decode — two different machines

Prefillread the whole prompt → first tokensets TTFT ··· Decodeone token at a time → each next tokensets TPOT / ITL
Prefill — compute-bound
  • Process all prompt tokens in parallel
  • Big matmuls saturate the GPU → FLOP-limited
  • Sets TTFT = time to first token
Decode — memory-bound
  • Generate one token per step, autoregressively
  • Tiny matmul, must re-read all weights → bandwidth-limited
  • Sets TPOT (a.k.a. ITL) = time per output token

TTFT = latency to the first token (prefill). TPOT/ITL = inter-token latency, the gap between each subsequent token (decode).

This is the most important distinction in the deck. Inference has two phases with opposite hardware characters. Prefill ingests the whole prompt at once — one big batched matmul over hundreds or thousands of tokens — so it saturates the GPU's compute units; it's FLOP-bound, and it determines TTFT, time-to-first-token. Decode then emits tokens one at a time: each step is a tiny matrix-vector product, but to produce that one token you must stream every weight of the model out of HBM. So decode is bandwidth-bound — you're waiting on memory, the compute units sit mostly idle — and it sets TPOT, the time per output token, also called inter-token latency or ITL. For a short answer TTFT dominates the felt experience; for a long generation TPOT dominates. Almost every technique later targets one phase or the other, so always ask "is this helping prefill or decode?"

Intuition

The note-taking analogy

  • To write the next word, the model must "look back" at every previous token.
  • Naively, that means re-reading the whole book from page one for each new word.
  • Instead: keep notes — a summary (keys & values) for every token you've already read.
  • Now each new word only needs the new token plus your notes — not a full re-read.
  • Those notes are the KV cache: re-reading is O(n) per token; reading notes is O(1).
Lean on this the whole "how it works" section. Attention means every new token attends to all previous tokens — so the dumb way to generate token number 500 is to re-run the model over all 499 earlier tokens, then 500 for token 501, and so on: quadratic, wasteful, re-reading the entire book to write each next word. The fix is to take notes as you read: for each token, once, compute and store its key and value vectors. Then for a new token you only compute its query, attend against the stored notes, and append your new note. The notes are the KV cache. Re-reading from scratch is O(n) work per token; consulting cached notes is O(1) per token. That single idea is why every serving system is built around the KV cache — and why its size is the thing that fills your GPU.

Intuition · the metrics picture

What the user actually feels

  • TPOT / ITL — the streaming "typing speed." Dominated by decode.

A picture of latency from the user's seat. Send a prompt and you stare at a spinner — that's TTFT, set by prefill plus any time the request waited in the scheduler queue. Then tokens stream in, and the gap between them is TPOT, also called inter-token latency, set by decode speed. A rough end-to-end estimate is TTFT plus N times TPOT for an N-token answer — useful but it can mislead, because it ignores queueing, batch interference, and variation. The subtle trap: streaming output makes a product feel fast even if the per-token rate is slow, so a great average TTFT can hide a sluggish trickle. This is exactly why we'll insist on measuring goodput under an SLO rather than a single headline number.

How it works · the KV cache

The KV-cache mechanism, step by step

One pass: prefill the prompt, cache K&V per layer/head, then decode token-by-token — the cache grows, but each step stays O(1).

Drive this off the widget and walk the loop. Prefill runs the entire prompt through the model in one parallel pass — that's the compute-bound phase that sets TTFT. As it goes, every token's key and value vectors are stored, for every layer and every head — that's the cache. Then decode begins: for each new token we compute only its query, attend it against the whole cached K and V, sample the next token, and append that token's own K and V back into the cache. The cache grows by one column per step, but the per-token work is constant — O(1) — instead of re-running the model over the whole prefix, which would be O(n). The loop arrow is the point: this repeats for every output token. The catch we'll quantify next: the cache grows linearly with sequence length and with batch size, and it lives in precious HBM.

How it works · cache fills cell-by-cell

Watch the cache grow as we decode

  1. Prefill fills the first column for every layer at once (the prompt).
  2. Decode t₂: attend over column 1, append a new column.
  3. Decode t₃: another column — cache grows linearly.
  4. Decode t₄: still O(1) work per token…
  5. …t₅ and on — the cache is the cost that grows.

With cache: O(1) compute per token, but memory grows O(n). Without: O(n) recompute every token → O(n²) total.

This is the figure to linger on. The grid is the KV cache: rows are a few transformer layers, columns are token positions, and each cell holds the stored key and value vectors for that token at that layer. Prefill lights up the first column for all layers simultaneously — the whole prompt processed in parallel. Then each decode step appends exactly one new column: attend the new token's query over all existing columns, then write its own K,V into the fresh column. Make the contrast explicit. With the cache, the compute per token is constant — O(1) — because you never recompute old tokens; the price you pay is memory that grows O(n) with sequence length. Without a cache you'd recompute the entire prefix at every step, which is O(n) per token and O(n-squared) over the whole generation — completely impractical. So the cache trades a manageable, linear memory cost for an enormous compute saving, and that memory growth is precisely why KV-cache size, not the weights, usually caps how many sequences you can serve at once.

How it works · continuous batching

Don't make finished requests wait

Static batching
  • Lock a batch, run until every sequence finishes.
  • Short replies sit idle waiting for the longest one — head-of-line blocking.
  • GPU runs at the speed of the slowest request.
Continuous (in-flight) batching
  • Schedule at the token level, not the request level.
  • A finished sequence leaves; a waiting one joins the very next step.
  • GPU stays full → far higher throughput, no idle slots.

Because decode re-reads the weights once per step, adding more sequences to that step is nearly free throughput — the whole reason batching wins.

Continuous batching is the single biggest throughput win in modern serving, and it falls straight out of "decode is bandwidth-bound." Static batching groups N requests, runs them lock-step, and won't release the batch until the longest sequence finishes — so a request that wanted ten tokens is held hostage by one that wanted a thousand. That's head-of-line blocking, and it leaves GPU slots idle. Continuous batching — also called in-flight or dynamic batching — schedules at the granularity of a single decode step: every step it evicts sequences that just finished and admits new ones from the queue, so the batch is continuously topped up and the GPU never idles. Why this is a free lunch: in decode you must stream all the model weights out of memory once per step regardless of batch size, so processing 32 sequences in that step costs almost the same memory traffic as processing one — you amortize the expensive weight read across many requests. That's the roofline intuition we'll formalize later: batching pushes decode from the memory-bound region toward compute-bound, raising utilization until the GPU's compute finally saturates.

Approaches · 1 of 5

The KV cache foundational

What: store the per-token Key and Value tensors from every layer so each new token only attends — never recomputes the whole prefix. How: during decode, append the new token's K,V to a growing buffer; attend the current query against all cached K,V. Solves: turns decode from O(n²) recompute into O(n) append — but the buffer is now the central memory consumer.

Pros
  • Each step is cheap — one new token, not the whole sequence
  • Makes long-context autoregressive decode tractable
Cons
  • Grows linearly with sequence length × batch — eats VRAM fast
  • Decode becomes memory-bandwidth bound (you read the whole cache each step)

vs baseline: naive decode re-runs attention over the full prefix every step; the cache trades compute for memory — and that memory is what every later technique fights to shrink.

Start here because the KV cache is the thing every other memory technique exists to attack. Without it, generating token number 1000 means re-running attention over the previous 999 tokens at every single step — quadratic, wasteful, and slow. The cache stores each layer's Keys and Values so the new query attends against a buffer that you just append to: O(n) work per step instead of O(n²). The cost is memory: that buffer grows with sequence length and with batch size, and on a 70B model at long context it can rival or exceed the weights. It also makes decode memory-bandwidth bound, because every step reads the entire cache back from HBM. Hold that fact — "decode is bandwidth bound" — it explains why a smaller cache speeds latency and why batching lifts utilization.

How it works · KV cache

KV cache — step by step

Prefillencode prompt → Fill cachestore K,V/layer → New tokencompute q,k,v → Append k,vgrow buffer ↩ Attend & emitloop
  1. Prefill: run the full prompt once, computing K,V for every token & layer (compute-bound).
  2. Store those K,V tensors in the cache — one slot per token, per layer.
  3. For each new token, compute its query, key, value.
  4. Append the new k,v to the buffer; the cache grows by one token.
  5. Attend the query against all cached K,V, emit the token, loop.

Size ≈ 2 × layers × kv_heads × head_dim × seq_len × bytes — the 2 is K and V.

Walk the two phases. Prefill processes the whole prompt in parallel — it is compute-bound and fills the cache in one big pass; this is what TTFT measures. Then decode is the loop: each new token contributes one k and one v that you append, and its query attends against everything cached so far. The formula on the slide is worth memorizing for interviews — two (for K and V), times layers, times KV heads, times head dim, times sequence length, times bytes per element. Plug in an 80-layer, 8-KV-head, 128-head-dim FP16 model at 8K tokens and you get roughly 2 × 80 × 8 × 128 × 8192 × 2 bytes ≈ 2.7 GB for a single request — per concurrent sequence. That number is why KV reduction and KV quantization matter, and it sets up everything in approach 5.

Approaches · 2 of 5

Quantization overview memory

What: store weights (and maybe activations) in low-bit integers/floats instead of FP16. How: map a float range to a grid via a scale (and zero-point); pick the granularity — per-tensor, per-channel, or per-group. Solves: INT4 weights are ¼ the size of FP16 — a 70B model drops from ~140 GB to ~35 GB, fitting cheaper GPUs and freeing room for KV cache.

Pros
  • Big memory cut; faster bandwidth-bound decode
  • Weight-only INT4 is near-lossless on perplexity
Cons
  • Activations have outlier features — huge values that wreck a shared scale
  • Aggressive quant can quietly break long-horizon / agentic tasks

vs previous: the KV cache shrank per-token state; quantization shrinks the weights — the other half of the VRAM bill.

Lay the vocabulary here because the next slides assume it. Two axes. First, what you quantize: weight-only (compress the stored weights, dequantize to FP16 to compute — INT4 GPTQ/AWQ live here) versus weight-and-activation (quantize both so the matmul itself runs in INT8 — that is what gives real compute speedups, and SmoothQuant lives here). INT8 dynamic quantization is the common W+A starting point. Second, when you quantize: PTQ (post-training, calibrate on a small dataset, cheap) versus QAT (quantization-aware training, simulate the rounding during training, expensive but recovers accuracy at very low bit-widths). And the granularity of the scale — per-tensor, per-channel, per-group — trades overhead for accuracy. The villain that motivates approach 3 is outlier features: a handful of activation channels carry values orders of magnitude larger than the rest, and a single shared scale either clips them or wastes all its precision on them.

How it works · Quantization

Quantization — step by step

FP16 weightsfull precision → Calibratefind ranges → Pick scaletensor/chan/group → Roundto INT grid → Store + dequantat compute
  1. Start from FP16 weights (and a small calibration set for activations).
  2. Calibrate: measure the value range per tensor / channel / group.
  3. Choose a scale (and zero-point) — finer granularity = better fit, more overhead.
  4. Round each value to the nearest grid point: q = round(w / scale).
  5. Store the ints; dequantize w ≈ q × scale only when computing.

Granularity: per-tensor (1 scale, cheap, brittle) → per-channel → per-group Q4_K-style (one scale per 32–128 weights, best accuracy/overhead balance).

The mechanism is a linear map: divide by a scale, round to the integer grid, store the int. At compute time you multiply back by the scale. The calibration step is where PTQ gets its ranges — feed a few hundred representative samples and watch the activation magnitudes; a poorly chosen calibration set (wrong domain, too short, all one language) gives bad ranges and silent quality loss. Granularity is the key knob: per-tensor uses one scale for the whole weight matrix — cheapest but a single outlier ruins everyone's precision. Per-channel gives each output channel its own scale. Per-group (block-wise, say one scale per 32 or 128 weights) is the modern default — it isolates outliers into small blocks and is what the GGUF K-quants use. The trade-off is the extra scales themselves cost a little memory and lookup, but the accuracy win is large.

Approaches · 3 of 5

GPTQ · AWQ · SmoothQuant methods

What: three PTQ recipes that each beat naive rounding by being smarter about error and outliers. How: GPTQ does error-compensated 2nd-order rounding; AWQ is activation-aware, protecting salient weight channels; SmoothQuant migrates activation outliers into the weights to make INT8 W+A viable. Solves: the outlier problem — the reason naive low-bit quant collapses.

Pros
  • GPTQ/AWQ: strong INT4 weight-only quality
  • SmoothQuant: enables fast INT8 weight+activation matmuls
Cons
  • GPTQ needs calibration + a second-order solve (slower to produce)
  • SmoothQuant needs a tuned migration strength α

vs previous: the overview did naive round-to-nearest; these three add error compensation, salience awareness, or outlier migration — that is what makes low-bit usable.

SmoothQuant ↗ · AWQ ↗ · GPTQ ↗

These three are the most-asked names in a quantization interview, so be crisp on the one-line idea for each. GPTQ (Frantar et al.) quantizes weights column by column and uses second-order (Hessian) information to compensate — after rounding one weight, it nudges the remaining unquantized weights to absorb the error, so the layer's output stays close. AWQ (Lin et al.) observes that not all weights matter equally: a small fraction of channels are "salient" because they multiply large activations; it scales those channels up before quantizing so they keep precision, leaving perplexity nearly untouched without any backprop. SmoothQuant (Xiao et al.) tackles the W+A case: activation outliers are the blocker for INT8 activations, so it mathematically migrates the difficulty from activations into weights — divide activations by a per-channel factor, multiply the weights by it — making both INT8-friendly. The knob is alpha, the migration strength. GPTQ and AWQ are weight-only INT4 workhorses; SmoothQuant is the one that unlocks genuinely faster INT8 compute.

How it works · the three methods

GPTQ · AWQ · SmoothQuant — step by step

Outlier problembig act values → GPTQcompensate error → AWQprotect salient W → SmoothQuantmigrate to W → Usable low-bitINT4 / INT8 W+A
  1. The problem: a few activation channels are huge — naive shared scales clip or waste precision.
  2. GPTQ — quantize weights column-by-column; use 2nd-order info to push rounding error onto not-yet-quantized weights.
  3. AWQ — find salient channels (big activations), scale them up to preserve their precision, then quantize.
  4. SmoothQuant — divide activations by per-channel s, multiply weights by s — outliers move into the easier-to-quantize weights.
  5. Result: INT4 weight-only (GPTQ/AWQ) or fast INT8 weight+activation (SmoothQuant).
Drive this as a contrast, not three separate stories. The shared villain is the outlier — step one. GPTQ's trick is sequential error compensation: it is greedy column-by-column rounding where each rounding decision is corrected by adjusting the weights it has not touched yet, guided by the inverse Hessian. AWQ's trick is asymmetry: instead of fixing all weights it identifies the roughly 1% of channels that ride on large activations and protects only those via a scaling reparameterization — cheap and no gradients. SmoothQuant's trick is algebraic identity: X·W is unchanged if you divide X by a per-channel s and multiply W by s, so you can drag the activation spikes into the weights where a per-channel scale handles them comfortably — this is the only one of the three aimed at making the activation matmul itself INT8. Point out the end states differ: GPTQ and AWQ leave you with INT4 weights and FP16 activations; SmoothQuant gives W8A8.

Approaches · 4 of 5

FP8 & formats numerics

What: floating-point (not integer) low precision — FP8 — plus the file/label formats that ship quantized models. How: FP8 has two flavors — E4M3 (4 exp, 3 mantissa: more precision, smaller range) and E5M2 (5 exp, 2 mantissa: more range, less precision); GGUF packages models with labels like Q4_K_M; KV cache itself can be FP8/INT8. Solves: outliers gracefully (floats have an exponent) — native on Hopper/Ada GPUs.

Pros
  • FP8 tolerates outliers far better than INT8 (dynamic range)
  • Hardware-native FP8 matmul on H100/Ada — real speedups
Cons
  • 8 bits, not 4 — less memory savings than INT4
  • FP8 KV cache can hurt long-context recall if too aggressive

vs previous: GPTQ/AWQ/SmoothQuant fight outliers with integer tricks; FP8 sidesteps them with an exponent — and needs the right silicon.

FP8 is the answer to "why not just use INT8 everywhere?" Integers have uniform spacing and no exponent, so outliers force a bad scale; floating point spends bits on an exponent and so spans a huge dynamic range, absorbing outliers naturally. The two FP8 layouts are a classic interview point: E4M3 uses four exponent bits and three mantissa bits — more mantissa means more precision but a smaller range, good for forward activations and weights; E5M2 uses five exponent and two mantissa — more range, less precision, historically used for gradients. The catch is FP8 is eight bits, so memory savings are roughly half FP16, not the quarter INT4 gives — you choose FP8 for speed and robustness, INT4 for raw footprint. Then formats: GGUF is the llama.cpp ecosystem's single-file format; a label like Q4_K_M means 4-bit, K-quant (block-wise mixed scheme), Medium size variant — the M/S/L pick how many sensitive tensors stay higher precision. Finally the cache: you can quantize the KV cache to FP8 or INT8 to roughly halve it, but be careful — over-quantizing keys and values degrades long-context retrieval, the kind of thing perplexity misses but a needle-in-haystack eval catches.

How it works · FP8 & formats

FP8, GGUF labels & KV quant — step by step

  1. Take an FP16 tensor — need range and precision in 8 bits.
  2. Choose the format: E4M3 (more mantissa → precision) vs E5M2 (more exponent → range).
  3. Run it through native FP8 tensor cores — real compute speedup, not just storage.
  4. For shipping: GGUF bundles weights; Q4_K_M = 4-bit, K-quant blocks, Medium variant.
  5. Optionally quantize the KV cache (FP8/INT8) to halve it — watch long-context recall.

Decode a label: Q=quantized · 4=bits · _K=block/K-quant · _M=size tier (S/M/L keep more sensitive tensors high).

Make the E4M3-vs-E5M2 choice tangible: same 8 bits, but you spend them on either mantissa (precision) or exponent (range). E4M3 is the default for inference weights and activations because the values are bounded and you want precision; E5M2 trades that for reach. The hardware point matters — FP8 only pays off on GPUs with FP8 tensor cores (Hopper H100, Ada); on older cards you get the memory but not the speed. Spend a moment decoding GGUF labels because students download these constantly: the letter Q, the bit-width, K for the newer block-wise K-quant scheme, and the S/M/L tier that decides how many attention/feed-forward tensors are kept at higher precision. End on KV-cache quantization as the bridge to approach 5 — it is the other way to shrink the cache, complementary to sharing heads, and its risk is silent degradation of long-context tasks.

Approaches · 5 of 5

GQA · MQA · MLA KV reduction

What: change attention so the KV cache itself is smaller. How: MQA shares one K/V head across all query heads; GQA shares K/V across groups of query heads (a middle ground); MLA compresses K/V into a low-rank latent and reconstructs on the fly. Solves: the cache's linear blowup — the central memory cost from approach 1.

Pros
  • MQA/GQA cut KV memory by the head-sharing ratio (e.g. 8× for 64→8 heads)
  • MLA: large compression with little quality loss (DeepSeek-V2/V3)
Cons
  • MQA can lose quality / training stability — GQA is the safe compromise
  • MLA adds reconstruction compute & complexity

vs previous: FP8/INT8 KV quant shrinks each cache entry; head-sharing & MLA shrink how many entries you store — they stack.

GQA ↗ · MQA ↗ · DeepSeek-V2 (MLA) ↗

This closes part A by attacking the cache's size directly. Recall the formula — the cache scales with the number of KV heads. Full multi-head attention gives every query head its own K and V head, which is great for quality but maximal for cache size. MQA (Shazeer) goes to the extreme: one shared K/V head for all queries, slashing the cache by the head count but sometimes hurting quality and stability. GQA (Ainslie et al.) is the now-standard compromise used in Llama-2/3 and most modern models: split query heads into a few groups, one K/V head per group — almost full-MHA quality at a fraction of the cache. MLA (DeepSeek-V2/V3) is the frontier: instead of sharing heads it projects K and V down to a small shared latent vector that you cache, then up-projects when attending — a big compression with minimal quality loss, at the cost of extra reconstruction compute and engineering complexity. The key framing: these reduce how many entries you store, while FP8 KV quant reduces the bytes per entry — orthogonal, so production stacks both.

How it works · GQA / MQA / MLA

GQA · MQA · MLA — step by step

MHAK/V per head → MQA1 shared K/V → GQAK/V per group → MLAcompress to latent → Smaller cachesame quality-ish
  1. Baseline MHA: every query head has its own K and V head — biggest cache.
  2. MQA: collapse to a single K/V head shared by all queries — smallest cache, some quality risk.
  3. GQA: group query heads, one K/V head per group — tunable sweet spot (e.g. 8 KV heads for 64 queries).
  4. MLA: project K,V into a low-rank latent; cache the latent, up-project at attention time.
  5. Net: cache shrinks by the sharing ratio (GQA/MQA) or by the compression rank (MLA).
Drive the meter as the cache shrinks left to right. The animation makes the evolution visible — the thing interviewers actually probe ("why GQA and not MQA?"). MHA is the full, expensive baseline. MQA is the aggressive extreme — one K/V head — which is why early adopters saw quality and training-instability issues. GQA is the Goldilocks answer that won: a handful of K/V head groups recovers nearly all the quality while still cutting the cache by the grouping ratio, which is why Llama-2 70B and Llama-3 use it. MLA is a different mechanism entirely — not sharing heads but low-rank compression: store a compact latent for K and V and reconstruct per-head at attention time, which is how DeepSeek-V2/V3 get a very small cache with minimal quality loss, paying in reconstruction FLOPs and implementation complexity. Close the section by reconnecting to the framing: head-sharing and MLA cut the number of stored entries; FP8/INT8 KV quant cuts the bytes per entry; quantizing the weights (approaches 2–4) cuts the model itself — three orthogonal levers that together decide how many concurrent sequences fit on the GPU.

FlashAttention IO-aware kernel

What: an exact attention kernel that is fast because it is memory-IO-aware, not because it does less math (Dao et al., 2022). How: tile Q/K/V into SRAM-sized blocks, compute attention block-by-block with an online softmax, and never write the full N×N score matrix to HBM. Solves: standard attention is bottlenecked by reading/writing that huge N×N matrix to slow HBM.

Pros
  • Exact (not an approximation) & much faster
  • Memory drops from O(N²) to O(N) — longer contexts fit
Cons
  • Hand-tuned CUDA, hardware-specific
  • Most win is in prefill; decode is bandwidth-bound elsewhere

vs previous: earlier work cut attention's FLOPs (sparse/linear); FlashAttention keeps exact math and instead cuts the slow HBM traffic.

Also — FlashInfer: a serving-oriented kernel dispatch library (2024) that extends FA to paged KV blocks, variable-length batches, MLA decompression, and sliding-window attention — the flexible attention backend used inside SGLang and vLLM.

Read more ↗

The key reframe for the interview: attention is not compute-bound, it is memory-bound. A naive kernel materializes the full N×N score matrix in HBM, writes it, reads it back for softmax, writes again — that round-tripping dominates the wall-clock. FlashAttention is described as IO-aware because it counts HBM reads/writes, not FLOPs, and minimizes them by keeping tiles in fast on-chip SRAM. It is mathematically exact — same output as vanilla attention — using an online (streaming) softmax that updates a running max and sum so it never needs the whole row at once. FA-2 improved parallelism and work partitioning; FA-3 (2024) targets Hopper with FP8 and async warp-group pipelining. Honest caveat: the big speedup is in prefill where N is large; single-token decode is bound by weight/KV bandwidth, which is what the other rungs attack.

How it works · FlashAttention

FlashAttention — step by step

Tile Q,K,VSRAM blocks → Block scoresQKᵀ in SRAM → Online softmaxrunning max/sum → Accumulate·V into output → Write onceO to HBM
  1. Split Q, K, V into blocks small enough to fit in fast on-chip SRAM.
  2. For each K/V block, compute that block's scores QKᵀ entirely in SRAM.
  3. Online softmax: keep a running max & denominator, rescale as new blocks arrive.
  4. Accumulate the weighted ·V into the running output — no full N×N matrix ever stored.
  5. Write only the final output O back to HBM (one pass, O(N) memory).
Walk the tiling loop concretely. The trick that makes it exact is the online softmax: as each new K/V block arrives you may discover a larger score than before, so you rescale the partial sums and the partial output by the ratio of the old to the new max — algebraically identical to computing softmax over the whole row at once. Because scores live and die in SRAM, you never pay the HBM cost of the N×N matrix, which is the whole game. The output is written exactly once. This is also why it enables longer contexts: peak memory is linear in sequence length, not quadratic.

PagedAttention / vLLM paging

What: manage the KV cache in fixed-size blocks via a block table, the way an OS manages virtual memory (Kwon et al., 2023). How: store each sequence's KV in non-contiguous physical blocks; a per-sequence block table maps logical positions to physical blocks; allocate blocks on demand. Solves: naive contiguous KV allocation wastes huge memory to internal & external fragmentation.

Pros
  • Near-zero fragmentation → far higher batch / throughput
  • Copy-on-write block sharing for parallel samples & beams
Cons
  • Custom attention kernel must gather scattered blocks
  • Block-table bookkeeping & small per-block overhead

vs previous: FlashAttention sped the kernel; PagedAttention fixes how KV memory is laid out so you can pack many more sequences.

Read more ↗

This is the idea behind vLLM and the reason it became the default serving engine. The problem: if you pre-reserve one contiguous slab of KV per request sized to max length, most of it sits empty (internal fragmentation) and the leftover gaps can't fit the next request (external fragmentation) — papers measured 60-80% waste. PagedAttention borrows the OS playbook: KV lives in fixed-size blocks (say 16 tokens), each sequence keeps a block table mapping its logical token positions to wherever the physical blocks happen to be, and blocks are handed out on demand. Fragmentation collapses to at most one partial block per sequence, so you fit many more concurrent requests and throughput jumps. The cost: the attention kernel must gather KV from scattered blocks, and you maintain the page tables — a worthwhile trade.

How it works · PagedAttention

PagedAttention — step by step

Logical KVper seq → Block tablemaps to physical → Physical blocksnon-contiguous → Share / CoWparallel samples
  1. Each sequence sees a contiguous logical KV — positions 0,1,2,…
  2. A per-sequence block table maps each logical block to a physical block.
  3. Physical blocks live anywhere in GPU memory — non-contiguous, allocated on demand.
  4. Samples sharing a prompt point to the same physical blocks; copy-on-write on divergence.
Drive the OS analogy off the widget. Logical KV is the virtual address space — each sequence thinks its tokens are laid out contiguously. The block table is the page table that translates logical block to physical block. Physical blocks are the page frames, scattered across HBM and allocated only as the sequence grows, which is why fragmentation nearly vanishes. The fourth step is the elegant payoff: for parallel sampling or beam search, all candidates share the prompt's physical blocks through their block tables, so the prompt KV is stored once; only when a sample writes a divergent token do you copy that one block — copy-on-write. That makes spawning many samples from a shared prompt almost free in memory.

Prefix / prompt caching reuse

What: cache the KV of a shared prefix (e.g. a long system prompt) and reuse it across requests instead of recomputing it. How: hash the prompt prefix; on a hit, skip prefill for those tokens and start decoding from the cached KV; RadixAttention indexes shared prefixes in a tree. Solves: recomputing the same system prompt on every request burns prefill compute & TTFT.

Pros
  • Cuts TTFT & prefill cost dramatically for shared prefixes
  • Composes with paging — cached blocks are just shared blocks
Cons
  • Cache eviction: hot prefixes can be dropped under pressure
  • Only helps the shared portion; unique suffix still prefilled

vs previous: paging shares KV within a request's samples; prefix caching shares KV across different requests.

Read more ↗

A natural extension of block sharing across request boundaries. Workloads with a long fixed preamble — a 2000-token system prompt, a few-shot exemplar block, a shared document in RAG — pay to prefill those identical tokens on every single request. Prefix caching keeps their KV around and, on a matching prefix, jumps straight to decoding the new suffix, slashing time-to-first-token. SGLang's RadixAttention generalizes this with a radix tree so many requests sharing partial prefixes all reuse the common portion; vLLM and commercial APIs expose automatic prefix caching too. The catch is eviction: cache space is finite, so under load a hot prefix may be evicted and have to be recomputed — you size and prioritize the cache, ideally pinning known-hot system prompts.

How it works · Prefix caching

Prefix caching — step by step

Hash prefixsystem prompt → Lookuphit? → Reuse KVskip prefill → Prefill suffixunique part → Decodelow TTFT
  1. Hash the incoming prompt's prefix (token-block granular).
  2. Look it up in the cache / radix tree — is this prefix already computed?
  3. On a hit, reuse the cached KV blocks & skip prefill for those tokens.
  4. Only the unique suffix gets a fresh prefill.
  5. Begin decoding immediately → sharply lower TTFT.
Mechanically, prefixes are matched at block granularity so partial overlaps still hit. On a hit you pay zero prefill for the shared tokens — their KV is already in HBM as shared paged blocks — and only the request-specific suffix is computed. The win is concentrated in TTFT, which is dominated by prefill; for a chat app with a fat system prompt or a RAG pipeline reusing the same retrieved context, this can be the single biggest latency lever. Watch the eviction policy: LRU is common, but you often want to pin known-hot prefixes so a burst of unique requests can't flush your most valuable cache entry.

SGLang & RadixAttention serving runtime

What: an inference serving runtime built around deep prefix reuse and structured generation. How: RadixAttention indexes KV blocks in a radix tree — any shared prefix of any length is found and its blocks reused; XGrammar precompiles JSON/grammar schemas to per-state token-ID bitmasks applied at near-zero per-step cost. Solves: hash-based caches miss partial or multi-path overlaps; naïve constrained decoding is O(vocab × states) per step.

Pros
  • Radix tree captures all shared prefix lengths — multi-turn, branching, RAG
  • Structured output (JSON / regex / grammar) at <3 % throughput overhead via XGrammar
  • Strong performance on agentic / multi-call workloads via program abstraction
Cons
  • Younger ecosystem than vLLM; some quant / adapter integrations still maturing
  • Radix-tree bookkeeping adds memory-accounting overhead vs. simple hash maps

vs vLLM prefix cache: vLLM hashes fixed-length block sequences; SGLang’s radix tree matches any prefix length and shares blocks across many co-tenant requests simultaneously.

SGLang paper ↗

SGLang (from the Lianmin Zheng group at Berkeley) started as a high-level language for multi-call LLM programs, but its serving backend became the main story. RadixAttention is the core insight: instead of hashing each fixed-size block to look up cached KV, it maintains a radix tree where each arc encodes a block of tokens and each node stores cached KV state. Two requests sharing the first 1000 tokens and diverging at token 1001 share every block up to the split — zero KV duplication regardless of block-size alignment. The win is greatest for multi-turn chat (context grows each turn), RAG with a shared retrieved document across user queries, and few-shot chains with a fixed preamble. XGrammar addresses structured output: the JSON schema is compiled to a push-down automaton; a precomputed GPU-resident bitmask of valid next-token IDs is stored for every reachable state. Each decode step is: state → bitmask_table[state] → logits[~mask] = −inf → sample → advance automaton. O(1) per step, runs as a GPU kernel. vLLM 0.6+ integrated XGrammar as its default structured-output backend too.

How it works · SGLang

SGLang — step by step

Requests arriveshared prefixes → Radix lookupany prefix length → Share KV blocksmatched prefix → XGrammar maskprecompiled bitmask → Decodeconstrained & fast
  1. Multiple requests arrive — chat turns, RAG queries — sharing prefixes of varying lengths.
  2. RadixAttention traverses the radix tree to find the longest shared KV prefix for each request.
  3. Matched KV blocks are shared in-place (no copy) — only unique suffixes get a fresh prefill.
  4. For structured output: current automaton state is looked up in the precomputed bitmask table (O(1), GPU-resident).
  5. Logits for invalid tokens set to −∞ before softmax — 100 % schema compliance, <3 % overhead.
Concrete example: a RAG pipeline embeds the same retrieved document (say 600 tokens) in every user query. With hash-based prefix caching (vLLM default) you need the prefix to align exactly to block boundaries — a 600-token prefix in a 16-token-block system hits at 592 tokens and misses the last 8. RadixAttention matches all 600 because the tree traversal stops exactly where tokens diverge, not at a block boundary. For structured output: the automaton for {"name": string, "score": number} has states like EXPECT_OPEN_BRACE, EXPECT_KEY, IN_STRING, etc. The precomputed bitmask for EXPECT_KEY has only the token IDs that produce '"name"' or '"score"' set valid — all 128K others are masked to −inf. The decode loop calls bitmask_lookup(state), applies it via a fused GPU kernel, samples, and advances the automaton — a pointer dereference and an element-wise operation, not a Python FSM traversal.

Continuous (in-flight) batching throughput

What: schedule the batch at the token level — add & remove sequences every decode step instead of fixing the batch up front (Yu et al., Orca, 2022). How: after each step, evict finished sequences and admit waiting ones into the freed slots, so the GPU is never idling on stragglers. Solves: static batching's head-of-line blocking — the whole batch waits for the longest sequence.

Pros
  • Big throughput & GPU-utilization win under mixed lengths
  • New requests don't wait for the current batch to finish
Cons
  • More complex scheduler & per-step memory accounting
  • A long prefill admitted mid-flight can still stall decodes (→ next rung)

vs previous: paging/prefix-caching freed memory; continuous batching uses that freed memory to keep the batch full every step.

Read more ↗

Static batching groups N requests, runs them to completion together, and only then starts the next batch — so a 20-token reply is held hostage by a 2000-token reply sharing the batch (head-of-line blocking), and the GPU runs at a fraction of capacity as sequences finish at different times. Continuous batching, introduced by Orca and now standard in vLLM/TGI, reschedules every decode step: the instant a sequence emits its EOS it leaves the batch and a queued request takes its slot. The batch stays dense, GPU utilization stays high, and tail latency improves because newcomers don't wait for the whole batch. It depends on paging to make admitting/evicting sequences cheap. The remaining wrinkle — a freshly admitted long prefill can monopolize a step and stall everyone's decode — is exactly what chunked prefill fixes next.

How it works · Continuous batching

Continuous batching — step by step

Active batchN sequences → Decode 1 stepall advance → Evict donehit EOS → Admit waitingfill slots ↺ Repeatstay full
  1. Start with N active sequences sharing the batch.
  2. Run one decode step — every active sequence emits one token.
  3. Any sequence that hit EOS leaves the batch immediately.
  4. Pull waiting requests from the queue into the freed slots.
  5. Loop — the batch refills each step, so the GPU never idles on stragglers.
The contrast to teach: in static batching the diagram would show empty slots opening up as short sequences finish and just staying empty until the longest one ends. Continuous batching backfills those slots the very next step. Because vLLM keeps each sequence's KV in paged blocks, eviction frees blocks instantly and admission allocates new ones, so the swap is cheap. The result is high, steady utilization across a workload with wildly varying output lengths — which is the normal case for chat. Note this is token-level (iteration-level) scheduling, the precise interview term that distinguishes it from request-level static batching.

Chunked prefill scheduling

What: split a long prefill into fixed-size token chunks and interleave them with ongoing decodes in the same batch (Agrawal et al., Sarathi, 2023). How: cap the tokens processed per step; a 8K prompt becomes many chunks, each step mixing one prefill chunk with everyone's decode tokens. Solves: one giant prefill monopolizing a step and stalling every other user's decode.

Pros
  • Smooths TPOT / inter-token latency under long prompts
  • Keeps decodes flowing → better p99, fuller batches
Cons
  • Chunk size is a tuning knob: too small hurts prefill efficiency
  • Slightly higher TTFT for the long prompt being chunked

vs previous: continuous batching admits requests freely; chunked prefill stops an admitted long prefill from blocking everyone's decode.

Read more ↗

The problem: prefill is compute-heavy and scales with prompt length, decode is one token. If you let a single 8K-token prefill run as one step, every other user's decode is frozen for the duration — a visible latency spike, the stall this rung kills. Chunked prefill (from the Sarathi line, now default in vLLM) caps how many tokens a step may process; a long prefill is broken into chunks of, say, 512 tokens, and each scheduler step packs one prefill chunk together with the decode tokens of all active sequences. Decodes keep flowing, inter-token latency stays smooth, and the batch stays full of useful work. The knob is the chunk / token-budget size: too large and you reintroduce the stall, too small and prefill loses its batching efficiency, so you tune it against your TTFT-versus-TPOT SLO.

How it works · Chunked prefill

Chunked prefill — step by step

Long prefill8K tokens → Split chunks512 each → Mix w/ decodetoken budget → Step1 chunk + decodes ↺ Next chunkno stall
  1. A long prompt arrives — naively it would be one huge prefill step.
  2. Split it into fixed-size chunks (e.g. 512 tokens each).
  3. Set a per-step token budget mixing one prefill chunk with active decodes.
  4. Each step processes one chunk plus everyone's decode token.
  5. Loop through chunks — decodes never stall waiting for the big prefill.
  6. Bonus — CUDA graphs: capture the static decode-step CUDA kernel sequence once and replay it, eliminating per-step CPU→GPU dispatch overhead (20–50 % latency saving at small batch sizes where kernel-launch cost dominates).
Picture the timeline: without chunking, a tall prefill bar blocks the lane and all the little decode ticks pile up behind it; with chunking, the prefill is sliced and each slice rides alongside the decode ticks so the lane keeps moving. The token budget per step is what couples prefill and decode into one balanced batch — you are filling the step's compute with a prefill chunk while still emitting a token for every active sequence. The trade is explicit: you slightly delay the long prompt's first token to protect everyone else's inter-token latency, which is usually the right call for an interactive multi-tenant service.

Speculative decoding latency

What: a cheap draft proposes several tokens, the big target model verifies them in one pass — output is provably identical to plain sampling (Leviathan et al.; Chen et al., 2023). How: draft k tokens autoregressively, run the target once over all k in parallel, accept the longest prefix that matches the target's distribution, resample the first reject. Solves: decode is sequential & memory-bound — you do one expensive forward per token even though the GPU is idle on compute.

Pros
  • Fewer sequential target passes → lower latency, exact output
  • Self-draft variants (Medusa, EAGLE) drop the separate draft model
Cons
  • Speedup hinges on the acceptance rate; a bad draft can slow you down
  • Stops helping at large batch — the GPU is already compute-saturated

vs previous: batching/scheduling raise throughput; speculative decoding attacks single-request latency by cutting sequential steps.

Read more ↗

The insight that makes this exact: decode is memory-bandwidth bound, so a single target forward pass over k candidate tokens costs almost the same as a pass over one token — the weights are streamed in either way. So let a small fast draft guess k tokens, then verify all k with one target pass. A clever accept/resample rule (rejection sampling against the ratio of target to draft probabilities) guarantees the accepted tokens are distributed exactly as if the target had sampled them — no quality loss, this is the headline. Acceptance rate is everything: if the draft agrees with the target most of the time you accept many tokens per pass and go several times faster; if it rarely agrees you pay for the draft and gain little, even slow down. Medusa adds extra prediction heads on the target itself; EAGLE drafts in feature space — both eliminate the separate draft model. And it fades at large batch sizes: once the batch already saturates compute, those parallel verification FLOPs are no longer free.

How it works · Speculative decoding

Speculative decoding — step by step

Draft ksmall model → Verifytarget, 1 pass → Accept prefixmatching tokens → Resample rejectfix divergence ↺ Loopk more
  1. Draft model autoregressively proposes k candidate tokens (cheap).
  2. Target runs ONE forward pass scoring all k in parallel.
  3. Accept the longest prefix consistent with the target's distribution.
  4. At the first mismatch, resample that token from the corrected distribution.
  5. Loop — accepted run length per cycle is set by the acceptance rate.
Drive the loop off the widget; the meter is the acceptance rate / effective speedup. Step one: the draft proposes, say, 4 tokens, paying 4 cheap forward passes. Step two: the target verifies all 4 in one pass — and because decode is bandwidth-bound, that one pass over 4 tokens costs about the same as a normal single-token pass. Step three: walk left to right accepting tokens while the draft's choice is consistent with the target; you might accept all 4 (huge win) or just 1. Step four is the part that guarantees correctness — at the first divergence you resample from an adjusted distribution, so the emitted sequence is distributed exactly like plain target sampling. Then loop. Emphasize the two failure modes: low acceptance (draft too weak or off-distribution) erodes or reverses the gain, and large batch sizes remove the "free verification" because the GPU is already compute-bound rather than bandwidth-bound.

Tensor / pipeline / expert parallelism scale-out

What: shard a model that won't fit (or is too slow) on one GPU across many. How: TP splits each layer's matrices across GPUs (all-reduce per layer); PP splits the layer depth into stages; EP places different MoE experts on different GPUs. Solves: models larger than one device's memory, and per-layer latency that one GPU can't hit.

Pros
  • Fit huge models; TP can cut per-token latency
  • EP scales MoE capacity without dense cost
Cons
  • TP needs fast interconnect (NVLink ≫ PCIe) or comms dominate
  • PP adds bubbles; EP suffers from expert load imbalance

vs previous: everything above optimized one GPU; here we scale out across GPUs when one device isn't enough.

Read more ↗

Three orthogonal ways to split. Tensor parallelism (Megatron-LM) shards the weight matrices inside each layer column- and row-wise across GPUs; each GPU computes its slice, then an all-reduce per layer stitches the partial results — this is the communication-heavy one, fine over NVLink, painful over PCIe. Pipeline parallelism assigns contiguous blocks of layers to different GPUs and streams microbatches through; comms are cheap (just activations at stage boundaries) but you get pipeline bubbles, especially at low batch. Expert parallelism is for MoE: experts are scattered across GPUs and tokens are routed to their experts, so a token only touches a couple of experts — but if routing is skewed, some GPUs sit idle while hot experts queue, hurting latency. The practical rule: TP within a fast NVLink node, PP across nodes, EP for MoE — and interconnect bandwidth is the deciding constraint for TP degree.

How it works · Parallelism

TP vs PP vs EP — step by step

One layertoo big/slow → TPsplit width + all-reduce → PPsplit depth into stages → EProute to experts → InterconnectNVLink vs PCIe
  1. A layer too big or too slow for one GPU.
  2. TP: shard the matmuls across GPUs, then all-reduce every layer to merge partials.
  3. PP: put different layer-ranges on different GPUs, stream microbatches stage to stage.
  4. EP: scatter MoE experts across GPUs; the gate routes each token to its top-k.
  5. Interconnect decides feasibility — NVLink makes TP cheap, PCIe makes it the bottleneck.
Make the communication picture concrete. TP's all-reduce fires twice per transformer layer (after the attention projection and after the MLP), so for an 80-layer model that's ~160 collective ops per token — over NVLink at hundreds of GB/s it's tolerable, over PCIe it can dominate the step, which is the classic "TP=8 barely faster, 8x cost" trap. PP moves only activations between stage boundaries so it's bandwidth-light, but a single in-flight request leaves stages idle (the bubble), so PP is for throughput via microbatching, not single-request latency. EP only activates a couple of experts per token so compute is sparse, but skewed routing concentrates tokens on a few GPUs and the slowest expert sets the step time — load balancing and capacity factors are the fix. Tie it back: pick the split that matches your interconnect topology and your latency-versus-capacity goal.

Disaggregated prefill/decode serving frontier

What: run prefill and decode on separate GPU pools, transferring the KV cache between them (Splitwise, DistServe, 2024). How: prefill workers do the compute-bound prompt pass & ship the resulting KV to decode workers, which do the bandwidth-bound token loop. Solves: prefill (compute-bound) and decode (memory-bound) fight for the same GPU and interfere with each other's SLOs.

Pros
  • Each pool tuned to its phase → isolates TTFT from TPOT
  • Independently scale & right-size prefill vs decode hardware
Cons
  • Must transfer KV between machines — bandwidth & latency cost
  • Extra scheduling, placement & KV-routing complexity

vs previous: chunked prefill interleaves the two phases on one GPU; disaggregation physically separates them onto specialized pools.

Read more ↗

The endgame of the prefill-versus-decode tension. Prefill is compute-bound and bursty; decode is memory-bandwidth-bound and steady; sharing one GPU means they interfere — a big prefill spikes everyone's TPOT, and chunked prefill only softens this. Disaggregation (Splitwise from Microsoft, DistServe) gives each phase its own pool: prefill machines optimized for compute throughput, decode machines optimized for memory bandwidth and large batches. You can scale them independently and hit tight TTFT and TPOT SLOs simultaneously. The new cost the interviewer will ask about: the KV cache produced by prefill must travel to the decode worker — that's potentially gigabytes per request over the interconnect, so you need fast links (NVLink/InfiniBand/RDMA), careful KV-transfer scheduling overlapped with compute, and placement that keeps the hop short. It's SOTA for large-scale, SLO-bound deployments where the transfer cost is worth the isolation.

Multi-adapter serving & S-LoRA adapter batching

What: serve hundreds of LoRA adapters from a single GPU without reloading the base model. How: base weights stay GPU-resident; low-rank matrices A, B (rank 8–64) are swapped from a GPU/CPU adapter pool; the scheduler co-batches requests targeting the same adapter. Solves: hosting N fine-tuned variants on N separate GPUs is prohibitive when N is large or demand is spiky.

Pros
  • Hundreds of adapters, one GPU — shared base amortises the full weight cost
  • Adapter swap adds only a rank-r correction matmul (rank ≪ hidden dim)
Cons
  • Long-tail adapters can’t be co-batched → throughput drops for rare adapters
  • Cold adapter page-in (CPU → GPU) adds latency spikes for uncached adapters

vs one-model-per-adapter: S-LoRA pages adapter matrices like OS pages — GPU-resident for hot adapters, offloaded to CPU for cold ones, same base model for all.

S-LoRA paper ↗

The business case: a platform where thousands of customers each fine-tune a shared base LLM. Each customer has a LoRA adapter — a pair of low-rank matrices A (d×r) and B (r×d) with r typically 8 to 64. The base weights dominate memory (70 GB for a 70B model); the adapter is tiny (a few hundred MB at most). S-LoRA (Sheng et al., 2023) keeps the base in GPU HBM and serves adapters from a pool: hot adapters GPU-resident, cold adapters on CPU or NVMe, paged in on demand. Batching works by adapter affinity: the scheduler groups requests for adapter A together in the same step to share their A/B matmuls, then requests for adapter B, and so on. The challenge is load imbalance: a popular adapter can batch 32 requests per step while a rare one runs a batch of 1, so scheduling must balance throughput and latency across adapters. vLLM and SGLang both support multi-LoRA; the interview term is "multi-adapter serving" or "dynamic adapter loading." The interview question (Q42) asks specifically about the batching and memory challenges — answer: (a) adapter affinity scheduling to co-batch, (b) cold-start paging latency, (c) per-step rank-r matmul kernel overhead at small batches.

How it works · S-LoRA

S-LoRA — step by step

N requestsK distinct adapters → Group by adapterco-batch same LoRA → Load A, BGPU pool / CPU page-in → Base + ΔWh ← h + x · A · B → Batch decodeshared base, unique Δ
  1. Requests arrive targeting K different LoRA adapters (customer A’s fine-tune, customer B’s, …).
  2. Scheduler groups requests by adapter ID — same-adapter requests share the A/B matmul in one step.
  3. Hot adapters are already in GPU memory; cold adapters are paged in from CPU (small tensors, fast transfer).
  4. Each decode step: shared base matmul Wx, then per-adapter correction h ← h + x · A · B (one extra fused rank-r kernel).
  5. One expensive base read amortised across the whole batch; rank-r adapter overhead is <1 % of the base matmul cost at typical ranks.
The key algebra: LoRA adds BAx to each linear layer output, where B is d×r and A is r×d. For the base linear y=Wx, the adapted output is y=Wx+BAx. Serving this in a batch means running Wx once for the whole batch and adding BAx with the appropriate (B,A) pair for each request. If two requests target the same adapter they share (B,A) — that's co-batching. The rank-r overhead: for r=16, d=4096, the correction matmul is (4096,16)×(16,4096) — roughly 1/256 the cost of the base (4096,4096) matmul, negligible per token. S-LoRA's custom CUDA kernel handles the batched-GEMM with adapter-index metadata so multiple (B,A) pairs are dispatched efficiently in one pass. Throughput is high for popular adapters (large same-adapter batches) and falls for rare adapters (batch size 1–2). The eviction policy for the adapter pool mirrors KV-cache eviction: LRU or frequency-weighted, with the same trade-off of cache space vs. cold-start frequency.

Structured output at the serving layer constrained generation

What: enforce JSON / regex / grammar constraints during sampling without quality loss. How: precompile the schema to a finite-state or push-down automaton; for each decode step, look up the current state’s precomputed bitmask of valid next-token IDs and set all others to −∞ before softmax. Solves: LLMs produce malformed JSON 5–15 % of the time; naïve runtime FSM traversal at O(vocab × states) per step becomes the serving bottleneck at scale.

Pros
  • 100 % schema compliance — zero output-parse failures in production
  • XGrammar precompiled bitmasks: <3 % throughput overhead vs. unconstrained decoding
Cons
  • Naïve per-step FSM traversal (Python/CPU) bottlenecks at high QPS — must precompile
  • Very complex grammars or huge schemas inflate automaton compile time per request

Tools: Outlines (mask-based, regular grammars), Guidance (interleaved control flow), XGrammar (compiled push-down automata — default in SGLang; integrated in vLLM 0.6+).

The throughput angle is what Q68 in the interview bank specifically probes. A naïve implementation calls a Python FSM at every decode step for every sequence in the batch: traverse grammar state, compute which 128K vocab tokens are valid next, build a logit mask, apply it. At batch 32 with a 128K vocab that is 4M Python-level operations per decode step — easily dwarfing the GPU compute time. The fix is precompilation: at request start (or server startup for cached schemas), XGrammar converts the grammar to a push-down automaton and stores a precomputed dense bitmask for every reachable state as a GPU tensor. The decode loop becomes: state_id → bitmask_table[state_id] → logits &= bitmask (GPU kernel) → sample → automaton_advance(token) → new state_id. Total overhead per step is one GPU gather and one element-wise op — under 3% in benchmarks. Outlines (2023) pioneered the mask-based approach for regular expressions; XGrammar (from the SGLang team, 2024) extended it to context-free grammars (JSON, SQL, tool-call schemas) using push-down automata. Both vLLM and SGLang now use XGrammar as their default structured-output backend.

Walk through a concrete example: schema is {"name": string, "score": number}. Automaton states: OPEN_BRACE → EXPECT_KEY → COLON → OPEN_QUOTE_VALUE → IN_STRING_OR_NUMBER → COMMA_OR_CLOSE → DONE. At EXPECT_KEY only token IDs that spell '"name"' or '"score"' are valid — all others get −inf. At IN_STRING only printable characters are valid, and a closing '"' transitions to the next state. The precomputed bitmask for each state is a uint8 tensor of shape (vocab_size÷8,) — for a 128K vocab that is just 16KB per state, negligible memory. The GPU masking kernel does a vectorised AND of the logits tensor with the bitmask. The automaton advance is a table lookup (integer → integer), also O(1). The contrast to post-processing (parse output, retry on failure): post-processing doesn't guarantee convergence, wastes compute on invalid prefixes, and inflates tail latency — inline masking costs almost nothing and guarantees valid output every time.

The math · (a) KV-cache bytes

How big is the KV cache?

For one sequence, the cached keys and values total:

$$ \text{bytes} = 2 \cdot L \cdot n_{kv} \cdot d_{head} \cdot S \cdot b $$

The leading 2 = one copy for K, one for V — you cache both.

\(L\) = number of transformer layers (each keeps its own KV); \(n_{kv}\) = number of KV heads (MHA: = query heads; GQA/MQA: fewer).

\(d_{head}\) = dimension per head; \(S\) = sequence length (tokens cached so far); \(b\) = bytes per element (FP16 → \(b=2\), FP8/INT8 → \(b=1\)).

Linear in \(S\) and in batch \(B\): total \(\times B\). This is per request — multiply by concurrency.

This is the most-asked formula in serving interviews, so I make every symbol concrete. The factor of 2 trips people up — it is not the FP16 byte count, it is because every layer stores both a key tensor and a value tensor. L is layers because attention is per-layer and each layer keeps its own cache. n_kv is the number of KV heads, and this is exactly the knob GQA and MQA turn: full multi-head attention has n_kv equal to the query-head count, MQA collapses it to 1, GQA picks something in between like 8. d_head is per-head dimension; note n_kv times d_head is the KV model width, not the full hidden size. S is how many tokens are cached, which grows every decode step, so the cache is small at the start of a generation and large at the end. b is bytes per element — the entire pitch of FP8 or INT8 KV cache is halving b. And the whole thing scales linearly with batch size, which is why concurrency, not weights, is what usually OOMs a server.

The math · (b) arithmetic intensity

Why decode is memory-bound

The roofline lens: compare work done per byte moved.

$$ I = \frac{\text{FLOPs}}{\text{bytes moved}} \qquad \text{throughput} = \min\!\big(P_{\text{peak}},\; I \cdot BW\big) $$

\(I\) = arithmetic intensity; \(P_{\text{peak}}\) = peak compute (FLOP/s); \(BW\) = memory bandwidth (bytes/s).

Decode, batch 1: read all weights to make one token → \(I\) tiny → you're stuck on the \(I\!\cdot\!BW\) (memory) side of the roofline.

Batching: reuse the same weights for \(B\) tokens → \(I \approx B\!\cdot\!(\text{ops/byte})\) rises, so \(I\!\cdot\!BW\) climbs until it hits \(P_{\text{peak}}\).

Once \(I \cdot BW \ge P_{\text{peak}}\) you're compute-bound → bigger batches stop helping latency. That knee is the sweet spot.

The roofline model is the cleanest way to explain why every decode trick exists. Arithmetic intensity is FLOPs per byte moved from HBM. At batch one, generating a single token requires reading every weight in the model from memory but doing only a handful of FLOPs with each — so intensity is tiny and you sit way over on the memory-bound left side of the roofline, where achievable throughput is bandwidth times intensity. The GPU's compute units are mostly idle; you are bottlenecked on moving weight bytes. Batching is the fix: when B sequences decode together, you read each weight once and reuse it for all B tokens, so intensity scales with B and the bandwidth-limited throughput climbs. This continues until intensity is high enough that you hit the flat peak-compute ceiling — the roofline knee — after which adding batch raises latency without raising throughput. That knee is exactly the batch size a good scheduler targets, and it's why prefill (which processes many tokens at once) is compute-bound while decode is memory-bound.

The math · (c) latency & speculative speedup

End-to-end latency, and the spec-decoding win

For a request that emits \(N\) tokens:

$$ T_{\text{total}} = \text{TTFT} + (N-1)\cdot \text{TPOT} $$

TTFT = time to first token (the prefill cost); TPOT = time per output token (each decode step). The \(N\!-\!1\) is because the first token is already in TTFT.

Short replies → TTFT dominates; long replies → the \((N\!-\!1)\!\cdot\!\text{TPOT}\) term dominates.

Speculative decoding: a draft proposes \(\gamma\) tokens, the target verifies them in one pass. With acceptance rate \(\alpha\), expected tokens generated per cycle is

$$ \mathbb{E}[\text{tokens/cycle}] = \frac{1-\alpha^{\gamma+1}}{1-\alpha} $$

Speedup ≈ that many target-steps' worth of tokens per verify pass. High \(\alpha\) → big win; low \(\alpha\) → the draft cost can make it a slowdown.

Two ideas here. First, the latency decomposition: total time is time-to-first-token plus per-output-token time times the remaining tokens. The minus one matters — the first token's cost is the prefill, which is TTFT, so you only pay TPOT for the other N minus one. The teaching point is that which term dominates flips with reply length: a one-line answer is all TTFT, a long essay is all TPOT, and that determines whether you optimize prefill or decode. The honest caveat the source asks for: this treats TPOT as constant, but it actually drifts up as the KV cache grows and varies with batch composition, so the formula is a planning estimate, not a guarantee. Second, speculative decoding: a cheap draft model guesses gamma tokens ahead and the expensive target verifies all of them in a single forward pass. With per-token acceptance probability alpha, the expected number of tokens accepted per verification cycle is that geometric-series formula — roughly 1 over 1 minus alpha for large gamma. Because the output is rejection-sampled to match the target's distribution, quality is unchanged. The whole thing only pays off when alpha is high enough that the extra accepted tokens outweigh the draft model's cost; at low acceptance, or at large batch where the GPU is already saturated, it becomes a net slowdown.

In code · the decode loop

Autoregressive decode with a growing KV cache

def decode(model, prompt_ids, max_new=128):
    # PREFILL: one pass over the whole prompt, returns past_kv
    logits, past_kv = model(prompt_ids, past_kv=None)
    next_id = logits[:, -1].argmax(-1, keepdim=True)
    out = [next_id]

    # DECODE: feed ONE token at a time, reuse + grow the cache
    for _ in range(max_new - 1):
        logits, past_kv = model(next_id, past_kv=past_kv)
        next_id = logits[:, -1].argmax(-1, keepdim=True)
        out.append(next_id)
        if next_id.item() == EOS:
            break
    return torch.cat(out, dim=-1)

class Attn:
    def forward(self, x, past_kv):
        k, v = self.k_proj(x), self.v_proj(x)   # only for the NEW token
        if past_kv is not None:
            pk, pv = past_kv
            k = torch.cat([pk, k], dim=1)        # concat past K
            v = torch.cat([pv, v], dim=1)        # concat past V
        out = attention(self.q_proj(x), k, v)    # attend over full cache
        return out, (k, v)                       # return grown cache
Walk the two phases. Prefill, lines one to six: the model sees the entire prompt in a single forward pass — this is compute-bound and its wall-clock is your TTFT — and it returns past_kv, the cached keys and values for every prompt token. We take the last position's logits and greedily pick the first generated token. Decode, lines eight to fourteen: now we feed exactly one token at a time. The crucial efficiency is that we pass past_kv back in, so the model never re-encodes earlier tokens; each step is one TPOT. The attention class at the bottom shows the actual mechanism: we project K and V only for the new token, then concat them onto the cached past along the sequence dimension, attend the new query over the full grown cache, and return the enlarged cache for next step. That concat is literally the KV cache growing by one row per step — which is exactly the S in the bytes formula incrementing. In production this concat is replaced by paged writes into pre-allocated blocks, but the logic is identical.

In code · absmax quantize

Tiny absmax INT8 quantize / dequantize

import torch

def absmax_quantize(w, bits=8):
    qmax = 2 ** (bits - 1) - 1          # 127 for int8
    scale = w.abs().max() / qmax        # one scale per tensor
    q = torch.round(w / scale).clamp(-qmax, qmax).to(torch.int8)
    return q, scale

def dequantize(q, scale):
    return q.to(torch.float16) * scale  # back to (lossy) float

# round-trip: store q (1 byte) + one scale, recover ~w
w = torch.randn(4096, 4096, dtype=torch.float16)
q, scale = absmax_quantize(w)           # 2x smaller than fp16
w_hat = dequantize(q, scale)
err = (w - w_hat).abs().mean()          # small if no big outliers
This is the simplest real quantizer and the mental model behind GPTQ and AWQ. Absmax: find the largest absolute value in the tensor, divide by qmax (127 for signed int8) to get a single scale, then map every weight to the nearest integer in minus-127 to 127. Storage drops from two bytes to one per weight, a two-times memory and bandwidth win — and bandwidth is what decode is bound on, so this is also a speed win. Dequantize just multiplies the integers back by the scale to recover an approximate float. The mean-error line is the teaching hook: error stays tiny unless the tensor has a few huge outliers, because one giant value blows up the scale and crushes the resolution for everything else. That outlier problem is precisely why activation quantization is hard and why SmoothQuant migrates outlier magnitude from activations into weights, and why real methods use per-channel or per-group scales instead of one scale per tensor.

Worked example · (1) KV-cache size

KV cache for a 70B model at 8K context

Plug a 70B-class config into \( \text{bytes} = 2 \cdot L \cdot n_{kv} \cdot d_{head} \cdot S \cdot b \):

  1. Config. \(L=80\) layers, \(n_{kv}=8\) KV heads (GQA), \(d_{head}=128\), \(S=8192\) tokens, FP16 → \(b=2\) bytes.
  2. Per-token, all layers. \(2 \cdot 80 \cdot 8 \cdot 128 = 163{,}840\) bytes/token (the K+V for one position across all layers).
  3. × sequence length. \(163{,}840 \cdot 8192 = 1{,}342{,}177{,}280\) bytes.
  4. × bytes/elt. \(\times 2 = 2{,}684{,}354{,}560\) bytes \(\approx\) 2.5 GiB (≈ 2.68 GB) per request.
  5. Punchline. One 8K request ≈ 2.5 GiB of KV. Just 10 concurrent requests ≈ 25 GB — KV, not weights, sets your concurrency ceiling.

Had this been full MHA (\(n_{kv}=64\)), it'd be \(8\times\) bigger ≈ 20 GB/req — that's why GQA exists.

Do this arithmetic live; interviewers love watching the units. Config first: 80 layers, 8 KV heads thanks to GQA, head dim 128, 8192 tokens, FP16 so 2 bytes. I deliberately compute it in two pieces to keep the factor of 2 honest. First, bytes per token across all layers: the leading 2 for K and V, times 80 layers, times 8 KV heads, times 128 head dim, equals 163,840 bytes for one token's worth of cache. Then multiply by the 8192 sequence positions to get about 1.34 billion, and finally by the 2 bytes per element to land at roughly 2.68 billion bytes, which is about 2.1 gigabytes for a single 8K request. The takeaway I hammer: weights are a fixed one-time cost, but KV scales with both context and concurrency, so ten simultaneous 8K users already need 21 gigabytes of KV cache on top of the model. And the closing line motivates the whole GQA section — if this were full multi-head attention with 64 KV heads the cache would be eight times larger, about 17 gigabytes per request, which would make long-context serving nearly impossible. That memory pressure is the single biggest reason modern models ship with GQA.

Worked example · (2) concurrency budget

How many 4K sequences fit on an 80 GB GPU?

A 13B model in FP16 on one 80 GB GPU — how much concurrency does the leftover memory buy?

  1. Weights. 13B params × 2 bytes (FP16) = 26 GB.
  2. Overhead. Reserve ~10 GB for activations, CUDA context, fragmentation & temp buffers.
  3. KV budget. \(80 - 26 - 10 = \) 44 GB left for the KV cache.
  4. KV per 4K seq. Say \(L=40,\ n_{kv}=8,\ d_{head}=128,\ b=2,\ S=4096\): \(2\cdot40\cdot8\cdot128\cdot4096\cdot2 \approx 0.69\) GB/seq.
  5. Concurrency. \(44 \div 0.69 \approx\) ~63 concurrent 4K sequences.

Quantize weights to INT4 (≈6.5 GB) → ~63.5 GB for KV → ~90+ sequences. Quantization buys concurrency, not just a smaller model.

This is the capacity-planning question, and the method matters more than the exact number. Start with weights: 13 billion parameters at 2 bytes each is 26 gigabytes — the rule of thumb is roughly 2 gigabytes per billion params in FP16. Then you must reserve overhead that beginners forget: activations, the CUDA context, allocator fragmentation, and temporary buffers, call it 10 gigabytes. That leaves 44 gigabytes of the 80 for KV cache. Now size one 4K-token sequence with a plausible 13B config — 40 layers, 8 KV heads, head dim 128, FP16, 4096 tokens — which works out to about 0.69 gigabytes per sequence. Divide 44 by 0.69 and you get roughly 63 concurrent 4K sequences. The closing insight is the one that surprises people: quantizing the weights to INT4 shrinks them to about 6.5 gigabytes, which frees nearly 20 more gigabytes for KV and pushes concurrency past 90 sequences. So quantization isn't only about fitting a bigger model — on a model that already fits, it directly buys you throughput by converting freed weight memory into more KV cache, more batch, and a higher roofline.

The mistake juniors make is bringing the offline config (huge batches) to a chat product and blowing the latency SLO.

Walk this as the answer to "given an SLO, how do you pick size/quant/batch/parallelism." Order matters. First the quality bar fixes the smallest acceptable model — never start big "to be safe," because every extra parameter costs latency. Then quantize (AWQ or FP8) to fit it on fewer GPUs; the second-order win is that smaller weights leave more HBM for KV cache, which is what actually limits concurrency. Parallelism is a fit tool, not a speed tool: a single GPU avoids the per-layer all-reduce of tensor parallelism, so only add TP (and only within an NVLink node) when the model plus its KV won't fit. Now the two SLO knobs. TPOT < 50 ms is a decode-latency budget — pick the largest batch that still clears it, because batch trades throughput for per-token latency. Pitfalls · misconceptions

People think X, but actually Y (1/2)

⚠ Pitfall
  • Think: the weights fit in HBM, so memory is fine. Actually: the KV cache grows with concurrency × context and OOMs under load — cap concurrent tokens, page KV, shrink it with GQA/KV-quant.
  • Think: tensor parallelism always cuts latency. Actually: TP adds an all-reduce per layer; at small-batch decode, comms dominate — TP=8 can give ≈ the same latency at 8× the cost. TP is for fitting a model, not speeding small-batch decode.
First two, both about confusing "fits" with "fast/safe." Number one is the single most common production incident: weights are 26 GB on an 80 GB card, looks comfortable at QPS=1, then it OOMs the moment real traffic arrives. The culprit is the KV cache — every concurrent sequence allocates KV proportional to its length, so the sum, not the weights, exhausts memory. The fix is admission control (cap max concurrent sequences / total tokens), PagedAttention to kill fragmentation, and GQA or KV-cache quantization to shrink per-sequence KV. Number two: people reach for tensor parallelism to "make it faster." But TP shards each matmul and needs an all-reduce after the row-parallel projections — about two per transformer block, every token, on the critical path. At small-batch decode the work is bandwidth/latency-bound and barely parallelizes, so you add communication overhead and get roughly the same latency while paying for 8 GPUs. TP's real job is capacity (fit a model that doesn't fit on one GPU) and compute-bound prefill, not low-batch decode latency.

Pitfalls · misconceptions

People think X, but actually Y (2/2)

⚠ Pitfall
  • Think: tokens/sec is the benchmark. Actually: raw tok/s hides batch size, input/output lengths, and SLO violations — report goodput under SLO, plus TTFT/TPOT p50/p95/p99 with the request distribution.
  • Think: INT4 perplexity barely moved, so it's safe. Actually: perplexity averages over tokens and misses rare decisive ones — agentic / long-context evals (exact JSON keys, function names, long-horizon consistency) can still collapse.
Two more, both about misleading metrics. Tokens/sec alone is meaningless for comparison: a giant batch inflates tok/s while p99 latency is dreadful, and the number says nothing about input-vs-output token mix, concurrency, or prompt/generation lengths. The honest report is goodput — throughput of requests that met the latency SLO — plus TTFT and TPOT percentiles (p50/p95/p99) and the exact request distribution, hardware, and precision, so it's apples-to-apples. The INT4 trap is subtler and a favorite interview probe: perplexity is an average next-token likelihood, insensitive to the rare tokens that decide a tool call — a JSON key, a function name, the one digit that breaks long-horizon consistency. Small per-token probability shifts compound over a long agentic trajectory, so a model that "looks fine" on perplexity can fail catastrophically on task-specific evals. Diagnose with real task evals, check error compounding over long trajectories, inspect which modules were quantized (lm_head and attention are sensitive), and keep critical tensors higher precision.

The serving-systems frontier

🚀 What's new
  • Memory is the OS problem. PagedAttention (vLLM) treats KV cache like virtual memory — blocks, block tables, copy-on-write sharing — near-eliminating fragmentation.
  • IO-aware kernels won. FlashAttention / FlashAttention-2 made exact attention memory-traffic-bound, not FLOP-bound.

  • Architectural KV cuts. DeepSeek-V2's MLA compresses K/V into a latent — MQA-level memory, more expressiveness.
  • Phase-aware scheduling. Chunked prefill (Sarathi-Serve) and prefill/decode disaggregation (DistServe) optimize for goodput, not raw tok/s.
  • Serving runtimes competed. SGLang’s RadixAttention (2023) generalises prefix caching to a radix tree — any shared prefix length hits; XGrammar (2024) makes structured output a throughput-safe first-class feature used by both SGLang and vLLM.
  • Adapter serving scaled. S-LoRA (2023) showed hundreds of LoRA adapters can share one GPU by paging low-rank A/B matrices and co-batching adapter-affine requests — eliminating the one-GPU-per-adapter tax.
Where the field is now — five threads. One: PagedAttention reframed KV memory as an OS paging problem; fixed-size blocks, a block table mapping logical to physical, and copy-on-write sharing for parallel sampling/beam search turned majority-waste allocation into near-zero fragmentation and is the foundation of vLLM. Two: FlashAttention and its successor made attention exact but IO-aware — tiling and online softmax keep intermediates in SRAM and never write the N×N matrix to HBM, so the win is reduced memory traffic, not fewer FLOPs. Three: speculative decoding went self-drafting — Medusa bolts extra prediction heads onto the target model and verifies a tree; EAGLE-2 drafts at the hidden-feature level with dynamic draft trees for higher acceptance — both eliminate hosting a separate draft model. Four: MLA in DeepSeek-V2 attacks KV cost in the architecture itself, compressing keys/values into a low-rank latent that's cached and up-projected at attention time — MQA-level memory but more expressive. Five: scheduling got phase-aware — chunked prefill (Sarathi-Serve) interleaves prefill chunks with decode to stop latency spikes, and DistServe disaggregates prefill and decode onto separate pools so each runs on ideal hardware, both chasing goodput rather than headline throughput.

Practice — click a question to reveal the answer

Full bank: 11 · Inference Optimization & Serving — from the answer key, condensed.

  • What is the KV cache, and what goes wrong at decode time without it? Easy▸ reveal
    Each new token attends to all previous tokens' keys and values; the KV cache stores per-layer K/V so they're computed once and reused. Without it, every decode step recomputes K/V for the whole prefix, making total generation cost quadratic in length. It caches attention K/V per layer — not the output logits.
  • Define TTFT and TPOT. Which dominates a long generation vs. a short one? Easy▸ reveal
    TTFT (time to first token) is request→first token, dominated by prefill. TPOT (time per output token / ITL) is steady-state decode latency. Long generations are TPOT-bound (TPOT × N dominates); short ones are TTFT-bound. Report both — one alone hides the other workload's pain.
  • Why is decode memory-bandwidth bound while prefill is compute-bound? Med▸ reveal
    Prefill processes all prompt tokens in parallel — big high-intensity matmuls that saturate compute. Decode does one tiny matmul per step but must stream the full weights from HBM, so bandwidth dominates. This is exactly why large batches help decode (reuse one weight read across many tokens).
  • Give the KV-cache size formula and estimate it for 80 layers, 8 KV heads, head_dim 128, FP16, 8K context. Hard▸ reveal
    bytes = 2 × layers × kv_heads × head_dim × seq_len × dtype_bytes. Per token/layer: 2×8×128×2 = 4 KB; ×80 layers = 320 KB/token; ×8192 ≈ 2.5 GB for one request. Use num_kv_heads (GQA), not query heads, and don't drop the ×2 for K and V.
  • How do GQA/MQA change KV-cache memory vs. MHA, and what's the trade-off? Med▸ reveal
    KV size scales with num_kv_heads. MQA shares one K/V across all query heads (cuts cache by the head count); GQA shares across groups (e.g. 8×). Big memory/bandwidth savings enable longer context/bigger batches, for a slight quality loss. Query heads & attention FLOPs are unchanged — only K/V heads shrink.
  • How does MLA attack KV-cache cost, and what does it trade? Hard▸ reveal
    Multi-head Latent Attention (DeepSeek) compresses K/V into a low-rank latent that's cached, then up-projects at attention time — MQA-level memory with more expressiveness than GQA/MQA. Trade: extra decompression compute per step and architectural complexity. It's baked into the architecture/pretraining — not a post-hoc serving toggle.
  • Explain PagedAttention and the OS analogy. What does it fix? Med▸ reveal
    Naive serving pre-allocates a contiguous max-length KV buffer per request — huge internal/external fragmentation. PagedAttention splits KV into fixed-size blocks mapped by a block table, allocated on demand. Logical positions = virtual addresses, blocks = pages, block table = page table; copy-on-write enables cheap prefix sharing.
  • What problem does FlashAttention solve, and why "IO-aware"? Med▸ reveal
    Naive attention materializes the full N×N matrix in HBM (memory-bound, O(N²) memory). FlashAttention tiles with online softmax, keeping intermediates in SRAM and never writing the full matrix. "IO-aware" because the win is reduced HBM traffic, not fewer FLOPs. It's exact attention — not an approximation.
  • What is chunked prefill, what does it solve, and what's the tuning knob? Med▸ reveal
    It splits a long prompt's prefill into token chunks across scheduler steps so prefill interleaves with ongoing decode, preventing a monolithic prefill from stalling everyone's decode (TPOT spikes). Knob: chunk size / per-step token budget — bigger favors prefill throughput, smaller protects decode latency. Same total work, just rescheduled — it doesn't reduce compute.
  • Define continuous batching and contrast it with static batching. Med▸ reveal
    Static batching runs a fixed group to completion before admitting new requests — slots idle as sequences finish (head-of-line blocking). Continuous (in-flight) batching schedules at the iteration/token level: a finished sequence's slot frees immediately and a queued request joins mid-flight, keeping the batch full and maximizing utilization.
  • Why does scaling TP=1→TP=8 barely cut latency while cost goes up 8×? Hard▸ reveal
    At small-batch decode the work is bandwidth/latency-bound, not compute-bound, so splitting the matmul 8 ways gives little speedup while adding an all-reduce per layer on the critical path — comms cancel the gain. TP is for fitting a model and for compute-bound prefill, not low-batch decode latency.
  • Why is speculative decoding's output identically distributed to the target, yet faster? Med▸ reveal
    A small draft proposes K tokens; the target verifies them in one parallel pass. A rejection-sampling rule accepts the longest correct prefix and resamples the first rejection from a corrected distribution — output equals the target's exactly. Faster because one bandwidth-bound target pass confirms multiple tokens nearly free. Low acceptance or a slow draft can make it slower.
  • Compare GPTQ and AWQ — core idea of each. Med▸ reveal
    Both are weight-only PTQ to ~4-bit. GPTQ uses approximate-Hessian (second-order) info to quantize column-by-column, adjusting remaining weights to compensate rounding error. AWQ finds salient channels via activation magnitudes and scales the ~1% that matter so uniform quantization preserves them. AWQ scales salient weights — it doesn't keep them in higher precision.
  • Serves fine at low QPS but OOMs under load though weights fit — cause and fix? Med▸ reveal
    KV-cache growth: many concurrent (and long) sequences each allocate KV, and the sum exceeds free HBM even though weights fit. Fix: admission control (cap concurrent sequences/tokens), PagedAttention, GQA/KV-quant to shrink per-sequence KV, preemption/offload, right-size batch. It's not the weights or a leak — it's KV scaling with concurrency × length.
  • Why is "tokens per second" misleading, and what should you report instead? Med▸ reveal
    Raw tok/s hides batch size, input/output length mix, concurrency, and whether the latency SLO was met — a huge batch inflates it while latency is terrible. Report goodput under SLO, TTFT/TPOT p50/p95/p99, the request distribution, hardware/precision, and cost-per-token for apples-to-apples.
  • What is FlashInfer and how does it differ from FlashAttention in scope? Med▸ reveal
    FlashAttention is a single unified attention kernel (IO-aware, exact, for contiguous QKV). FlashInfer is a serving-oriented kernel dispatch library that provides the right attention kernel for each serving scenario: paged KV blocks, variable-length batches, MLA decompression, sliding-window — the flexible attention backend inside SGLang and vLLM. FA = the kernel; FlashInfer = the dispatch layer that calls the right variant per request type.
  • What do CUDA graphs and torch.compile contribute to decode-step latency, and why does it matter more at small batch? Med▸ reveal
    At small batch, GPU compute finishes quickly but CPU overhead (kernel launch, scheduler dispatch, Python loop) dominates decode latency. CUDA graphs capture the static kernel sequence in one graph object and replay it in a single launch, eliminating per-step CPU→GPU overhead (20–50 % saving at batch 1–8). torch.compile fuses Python/ATen ops. Both help most where compute is cheap and dispatch is expensive. At large batch the GPU is compute-saturated and launch overhead is negligible — the gain vanishes.
  • How do you serve hundreds of LoRA adapters on one GPU, and what batching / memory challenges arise? Hard▸ reveal
    Keep base model resident; maintain a GPU/CPU adapter pool for low-rank A, B matrices. Co-batch requests targeting the same adapter to amortise the rank-r matmul. Challenges: (a) long-tail adapters can’t be co-batched → low throughput for rare adapters; (b) cold adapters must page in from CPU → latency spikes; (c) per-step rank-r matmul needs a custom batched GEMM kernel. The rank-r cost is tiny (1/rank² of base layer) — the bottleneck is adapter scheduling and cold-start paging, not the math.
  • What is the throughput cost of structured output, and how do XGrammar / Outlines avoid the bottleneck? Med▸ reveal
    Naïve: traverse the FSM at every decode step for every batch item → O(vocab × states) Python overhead per step, bottlenecking at high QPS. Fix: precompile the grammar to a push-down automaton at request start; store a GPU-resident bitmask of valid next-token IDs per state; decode step = bitmask lookup (O(1)) + GPU mask apply → <3 % throughput overhead. The bottleneck is the per-step FSM traversal, not the masking — precompilation is the entire fix.

Takeaways

What to remember

  • Two phases, two bottlenecks. Prefill is compute-bound (TTFT); decode is memory-bandwidth-bound (TPOT). Every tuning decision flows from this.
  • KV cache is the real memory limit. Weights fitting ≠ safe — concurrency × context OOMs you. Page it, shrink it (GQA/MLA/KV-quant), cap it.
  • Batching is the throughput lever for decode — continuous batching keeps the GPU full; size batch to the TPOT budget, not to peak tok/s.
  • Quantize weights to fit & free KV (AWQ/GPTQ/FP8) — but validate with task evals, not just perplexity.
  • Speculation & TP are conditional. Speculative decoding pays off at low batch; TP is for fitting, not small-batch speed.
  • Report goodput under SLO — TTFT/TPOT percentiles with the request distribution, never raw tokens/sec.
The six-point recap — the spine of the whole module. One: internalize the two-phase model; prefill compute-bound (drives TTFT, wants FP8/compute and chunking), decode bandwidth-bound (drives TPOT, wants big batches, GQA, quantized weights). Two: the KV cache, not the weights, is what OOMs you under load — page it with PagedAttention, shrink per-sequence cost with GQA/MLA/KV-quant, and cap concurrency with admission control. Three: batching is the decode throughput lever, but size it to the latency SLO, not to peak tokens/sec. Four: quantization fits the model and frees HBM for KV, but perplexity is a weak gate — validate on real tasks. Five: speculative decoding and tensor parallelism are conditional tools — speculation wins at low batch where there's spare compute; TP is for capacity and compute-bound prefill, not low-batch decode latency. Six: the honest benchmark is goodput under SLO with percentile latencies and the request distribution. If they remember two sentences: "prefill compute-bound, decode bandwidth-bound," and "the KV cache is what OOMs you, not the weights."
