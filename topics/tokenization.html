Tokenization & Embeddings

How raw text becomes a list of integer IDs, and how those IDs become the vectors a Transformer actually reads — the foundation layer, classic → SOTA.

📋 What you’ll learn (~12 min)

The tokenizer pipeline · every subword algorithm (BPE, byte-level, WordPiece, Unigram, SentencePiece) and the tokenizer-free frontier · static vs contextual embeddings · the embedding matrix & weight tying · a live tokenizer demo · and the pitfalls that bite (the “strawberry” problem).

1. Intuition

Tokenization is like writing with LEGO bricks instead of whole pre-built houses. Whole-word “houses” mean you need a brick for every word that exists (millions, and you still miss new ones). Single-letter bricks are universal but you need hundreds of them to build anything. Subwords are the sweet spot: a few common chunks (token, ##ization, ▁the) compose into any word, even ones never seen in training.

Embeddings turn each brick into a point in a high-dimensional “map of meaning,” where nearness encodes similarity: cat and kitten land close together, cat and thermodynamics far apart. But a static map can’t tell which sense of bank you mean — that takes context, which is the job of attention (next page).

2. How it works

2.1 — Text → tokens → IDs

The tokenizer is a deterministic pre-processor run before the neural network. A modern pipeline has four stages: normalization (e.g. Unicode NFC/NFKC, optional lowercasing), pre-tokenization (split on whitespace/punctuation so merges never cross word boundaries), the subword model (BPE / WordPiece / Unigram applied to each chunk), and post-processing (add special tokens like <BOS>/<EOS>, build the integer ID list). The output is a list of integers — the only thing the network ever sees. Note that leading spaces matter: GPT-style byte-level tokenizers treat "▁the" (space + the) and "the" as different tokens, which is why stray spaces in prompts and stop sequences trip people up.

2.2 — IDs → vectors (the embedding layer)

The embedding matrix E has shape (V, d): one row of length d (the model dimension) per vocabulary entry V. “Looking up” token ID i is just selecting row i — mathematically identical to multiplying a one-hot vector by E, but done as an indexed read so it’s cheap. For a batch of token IDs of shape (B, n) you get hidden states of shape (B, n, d). These vectors are learned, so similar tokens drift to similar regions of space. Many models tie this matrix with the output (unembedding) layer to save V·d parameters — which for a small model can be a large fraction of the total.

3. The approaches — full landscape

Every major tokenizer and embedding type, roughly classic → SOTA, with what it is, how it works, what it solves, and trade-offs. Each has a “read more” deep-dive link.

Word-level tokenization classic

What: one token per whitespace/punctuation word. How: build a fixed vocabulary of the most frequent words; everything else maps to <UNK>. Solves: short sequences, intuitive units.

Pros
  • Few tokens per sentence
  • Human-readable units
Cons
  • Huge vocab, still misses rare/new words (OOV)
  • No sharing between run/running/ran
  • Terrible for morphologically rich languages

Read more: Summary of the tokenizers · Hugging Face

Character / byte-level classic

What: one token per character (or per raw byte). How: tiny fixed alphabet (~100 chars, or exactly 256 bytes). Solves: zero OOV — anything can be spelled out; trivially multilingual.

Pros
  • No OOV ever; tiny vocab
  • Robust to typos, code, emoji
Cons
  • Very long sequences → attention cost (n²) explodes
  • Model must relearn spelling; weaker semantics per token

Read more: Character- & byte-level tokenization · Hugging Face

Byte-Pair Encoding (BPE) modern · dominant

What: the subword workhorse (Sennrich et al., 2015). How: start from characters; repeatedly find the most frequent adjacent pair in the corpus and merge it into a new symbol; stop at the target vocab size. At inference, apply the learned merges in order. Solves: the OOV/sequence-length trade-off — frequent words become one token, rare words fall back to a few subwords.

Pros
  • Open-vocabulary; tunable vocab size
  • Simple, fast, deterministic
Cons
  • Greedy merges aren’t globally optimal
  • Segmentation can be linguistically odd

Read more: Byte-Pair Encoding, step by step · Hugging Face NLP Course

Byte-level BPE SOTA default

What: BPE applied over the 256 raw bytes instead of Unicode characters (GPT-2 and most LLMs). How: the base alphabet is bytes, so any Unicode string is representable; merges build up common byte sequences. Solves: guarantees no <UNK> for any input, in any language or binary, with a compact base.

Pros
  • Truly universal coverage, never breaks
  • De-facto standard (GPT, LLaMA via SP-BPE)
Cons
  • A single emoji/CJK char can cost several byte-tokens
  • Multibyte chars can be split mid-character

Read more: Byte-level BPE (GPT-2 style) · Hugging Face NLP Course

WordPiece modern

What: BERT’s tokenizer. How: like BPE, but instead of merging the most frequent pair it merges the pair that most increases the likelihood of the training corpus under a unigram LM (score ≈ freq(ab)/(freq(a)·freq(b))). Continuation pieces are marked with ##. Solves: slightly more principled merges than raw frequency.

Pros
  • Likelihood-driven, good for encoders
  • ## makes word boundaries explicit
Cons
  • Still greedy; tied to its training corpus
  • Needs pre-tokenization

Read more: WordPiece tokenization · Hugging Face NLP Course

Unigram LM modern

What: a probabilistic tokenizer (Kudo, 2018). How: start with a large candidate vocabulary and prune it: keep a unigram probability per subword, and iteratively (via EM) drop the pieces that hurt corpus likelihood least. At inference, pick the highest-probability segmentation with Viterbi. Solves: a global probabilistic objective and multiple possible segmentations.

Pros
  • Often cleaner morphological splits than BPE
  • Enables subword regularization (sampling)
Cons
  • More complex training
  • Probabilities tied to corpus distribution

Read more: Unigram tokenization · Hugging Face NLP Course

SentencePiece framework

What: not an algorithm but a language-agnostic implementation (Kudo & Richardson, 2018) that runs BPE or Unigram. How: consumes raw text (no pre-tokenization), treating whitespace as a normal symbol it encodes as ▁. Solves: fully reversible, reproducible tokenization for any language — crucial for non-space-delimited languages (Chinese, Japanese).

Pros
  • Detokenization is exact (no lost spaces)
  • Works on any script, end-to-end
Cons
  • The ▁ meta-symbol surprises newcomers
  • Inherits whatever model (BPE/Unigram) you pick

Read more: SentencePiece · GitHub · google

Subword regularization / BPE-dropout robustness

What: sample different segmentations of the same string during training. How: Unigram sampling or randomly skipping BPE merges exposes the model to many tokenizations of one word. Solves: over-reliance on one brittle segmentation; acts as data augmentation, improving robustness (especially for MT and noisy text).

Pros
  • More robust, better generalization
  • Nearly free to add
Cons
  • Slightly slower training
  • Inference still uses the single best split

Read more: Subword regularization & BPE-dropout · SentencePiece

Tokenizer-free / byte-latent models frontier

What: drop the fixed vocabulary entirely and model raw bytes/characters (MEGABYTE, CANINE, Meta’s Byte Latent Transformer, T-FREE). How: group bytes into patches — fixed-size or, in BLT, entropy-based dynamic patches — and process them hierarchically, spending more compute where the text is less predictable. Solves: tokenizer brittleness — glitch tokens, multilingual unfairness, numeric/code splitting, and the “strawberry has how many r’s” character-blindness.

Pros
  • No tokenizer to train or mis-load; robust & fair across languages
  • BLT matches token-based LLMs at scale with better inference efficiency
Cons
  • Longer base sequences need clever hierarchies
  • Newer, less battle-tested than BPE

Read more: Byte Latent Transformer, explained · AI Papers Academy

Embeddings: from static to contextual

Static word embeddings — word2vec / GloVe / fastText classic

What: one fixed vector per word, learned from co-occurrence. word2vec (Mikolov 2013) predicts a word from its context (CBOW) or context from the word (Skip-gram), trained efficiently with negative sampling. GloVe (Pennington 2014) factorizes a global co-occurrence count matrix. fastText (Bojanowski 2016) sums character-n-gram vectors so it can embed unseen words. Solves: dense, semantic vectors where geometry ≈ meaning (the famous king − man + woman ≈ queen).

Pros
  • Cheap, interpretable, reusable
  • fastText handles OOV / morphology
Cons
  • One vector per word → can’t separate the two senses of bank
  • No context, no order

Read more: The Illustrated Word2vec · Jay Alammar

Contextual embeddings (the Transformer hidden state) SOTA

What: the vector a Transformer produces for a token depends on the whole sentence. How: attention blends in surrounding tokens, so bank in “river bank” and “bank account” get different vectors. Solves: polysemy — the core limitation of static embeddings. Sentence-embedding models (and Matryoshka representations, which nest usable shorter vectors inside longer ones) build on this for retrieval.

Pros
  • Sense- and context-aware
  • Backbone of RAG, search, reranking
Cons
  • Needs a full forward pass to compute
  • Anisotropy can distort naive cosine similarity

Read more: The Illustrated BERT, ELMo & co. · Jay Alammar

5. The math

Embedding lookup

For token ID \( t \) and embedding matrix \( E \in \mathbb{R}^{V \times d} \):

$$ \mathbf{x}_t = E_{t,:} = \mathbf{e}_t^{\top} E $$

V = vocabulary size, d = model dimension, \(\mathbf{e}_t\) = one-hot vector for ID \(t\). The right form shows why a lookup equals a one-hot matrix multiply — we just skip the multiply.

Embedding-layer parameters & tying

$$ P_{\text{embed}} = V \cdot d, \qquad P_{\text{tied}} = V \cdot d \;\;(\text{shared in + out}) $$

With V=32k and d=768 that’s ~25M parameters per matrix — for a small model, a big slice of the budget. Weight tying reuses one matrix for both input embedding and output projection, halving this cost and often improving quality.

6. In code — one BPE merge step

BPE’s whole training loop is “count adjacent pairs, merge the most frequent, repeat.” The core step:

python
from collections import Counter

# words as space-separated symbols; '_' marks end-of-word. value = corpus frequency
vocab = {'h u g _': 10, 'p u g _': 5, 'h u g s _': 5}

def best_pair(vocab):
    pairs = Counter()
    for word, freq in vocab.items():
        symbols = word.split()
        for a, b in zip(symbols, symbols[1:]):
            pairs[(a, b)] += freq
    return pairs.most_common(1)[0]

print(best_pair(vocab))   # -> (('u', 'g'), 20)  --> merge 'u','g' into 'ug' first

7. Worked example — tokenize & embed

Take “the cat sat.” A byte-level BPE tokenizer (with leading spaces attached) might emit three tokens ▁the, ▁cat, ▁sat → IDs [464, 3797, 3332]. The embedding lookup grabs rows 464, 3797, 3332 of E, giving three d-dim vectors (a (3, d) tensor). A position signal is added, and those vectors flow into the first attention layer (next page). Notice: “the” written with vs without a leading space are different IDs — a classic prompt bug. And a rarer word like “tokenization” would split into several pieces (token + ization), so token count ≠ word count.

8. When to use what — choosing a tokenizer

  • General LLM, many languages → byte-level BPE (or SentencePiece-Unigram). It’s the safe default.
  • Heavy code / structured data → train BPE on that data, reserve tokens for indentation/operators; check token fertility (tokens-per-word).
  • Robustness, morphology-rich or noisy text, fairness across scripts → consider char/byte or tokenizer-free (BLT).
  • Arithmetic-critical → digit-level or single-token numeric schemes (xVal/FoNE) to avoid inconsistent number splits.

9. Common pitfalls & misconceptions

  • “A token is a word.” No — one word is often several tokens (unhappiness → un+happi+ness), and a leading space is part of the token. This is why models miscount letters in “strawberry”: they never see the letters, only subword chunks.
  • “Byte-level BPE still needs UNK.” It doesn’t — every byte is in the base vocab, so any input is representable. UNK lingers only for legacy/compat reasons.
  • “Static embeddings capture meaning fully.” One vector per word can’t separate the senses of bank; only contextual hidden states can.
  • “Same text costs the same tokens in every model.” Different tokenizers → different counts → different cost and context usage; English-centric tokenizers make other languages more expensive.
  • “Glitch tokens are harmless.” Under-trained tokens (e.g. SolidGoldMagikarp) arise from tokenizer/training-data mismatch and cause bizarre behavior.

10. 🚀 What’s new in the industry

🚀 Beyond fixed-vocabulary BPE
  • Tokenizer-free / byte-latent. Meta’s Byte Latent Transformer groups bytes into entropy-based dynamic patches and matches token-based LLMs at scale — no tokenizer to train or mis-load, and no glitch tokens.
  • “Superword” tokens. SuperBPE lets tokens span whitespace, cutting token counts up to ~33% and improving downstream accuracy over same-size BPE.
  • Better numbers. xVal and FoNE encode each number as a single value-aware token, fixing the inconsistent digit-splitting that hurts arithmetic.
  • Adaptive embeddings. Matryoshka representations nest usable shorter vectors inside longer ones, so one embedding serves many storage/latency budgets — now standard in retrieval.

12. Key takeaways

  • Subwords win. Byte-level BPE / SentencePiece balance vocabulary size against sequence length and kill OOV.
  • A “token” is not a word, and a leading space is part of the token — the root of the “strawberry” miscount and many prompt bugs.
  • Embedding = a learned (V, d) lookup table; tying input/output saves V·d parameters.
  • Static embeddings give geometry; attention gives context. Only contextual hidden states disambiguate bank.
  • The frontier is going tokenizer-free (BLT) and smarter (SuperBPE, numeric schemes).
